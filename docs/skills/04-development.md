# Phase 4: Development

アルゴリズム設計を、実装可能な設計書と実装計画に落とし込むフェーズです。

## research-07-implementation

| 項目 | 内容 |
| --- | --- |
| コマンド | `/research-07-implementation` |
| 使用モデル | Sonnet |
| 目的 | 実装計画・アーキテクチャ・モジュール構成・テスト計画を作成する |

### いつ使うか

- 「実装計画を立てたい」「アーキテクチャを設計したい」「実験コードの設計」
  「テスト計画を作りたい」
- `research-06-algorithm-design` の次のステップとして

### 主な処理

1. **全体アーキテクチャ**: ディレクトリ構成、モジュール分割(擬似コードのアルゴリズム単位と
   1:1 対応させる)、設定管理・実験管理方針を設計する。
2. **詳細設計**: 主要クラス・関数のシグネチャ、データ構造、学習ループ・推論パイプラインを設計する。
   提案手法とベースラインを同一の評価コードで測れるインターフェースにする。
3. **テスト計画**: ユニットテスト(数式どおりの動作検証)、形状・数値健全性テスト、統合テスト、
   再現性テスト、過学習サニティチェックを設計する。
4. **CI・実装順序**: MVP → ベースライン再現 → 提案手法フル → アブレーション対応、の順で
   マイルストーンと完了条件を定める。

### 出力

```
research/
    implementation-plan.md  # マイルストーン、優先順位、実験管理方針、リスク
    architecture.md         # ディレクトリ構成、モジュール設計、クラス/API設計、データ構造
    test-plan.md            # テスト一覧(目的・入力・期待値)、CI設計
```

### 品質基準

- 「ベースラインを先に動かす」を原則とする(提案手法の評価はベースラインの再現ができてから)
- 実験の設定・結果が後から追跡できる設計にする(再現性フェーズを楽にする)
- 評価に関わる部分(指標計算・データ分割)のテストは必須とする

### 実装コードの独立レビュー

この設計書をもとに実際の実験コードを書き始める場合、書いたそばから自己判断だけで正しさを
確認しない。`research-08-evaluation/references/code-review.md` にある、過去の会話コンテキストを
渡さないサブエージェントによる3段階レビュー(設計適合・バグ → ダミーデータでの妥当性検証 →
実データ適用後の結果妥当性)を、test-plan.md のテスト計画と合わせて適用する。

# Phase 4: Development

アルゴリズム設計を、実装可能な設計書と実装計画に落とし込むフェーズです。

## research-07-implementation

| 項目 | 内容 |
| --- | --- |
| コマンド | `/research-07-implementation` |
| 使用モデル | Sonnet |
| 目的 | 実装計画・アーキテクチャ・モジュール構成・テスト計画・タスク分解を作成する |

### いつ使うか

- 「実装計画を立てたい」「アーキテクチャを設計したい」「実験コードの設計」
  「テスト計画を作りたい」「タスクに分解したい」
- `research-06-algorithm-design` の次のステップとして

### 主な処理

1. **全体アーキテクチャ**: ディレクトリ構成、モジュール分割(擬似コードのアルゴリズム単位と
   1:1 対応させる)、設定管理・実験管理方針を設計する。
2. **詳細設計**: 主要クラス・関数のシグネチャ、データ構造、学習ループ・推論パイプラインを設計する。
   提案手法とベースラインを同一の評価コードで測れるインターフェースにする。
3. **テスト計画**: ユニットテスト(数式どおりの動作検証)、形状・数値健全性テスト、統合テスト、
   再現性テスト、過学習サニティチェックを設計する。
4. **CI・実装順序**: MVP → ベースライン再現 → 提案手法フル → アブレーション対応、の順で
   マイルストーンと完了条件を定める。
5. **タスク分解**: 各マイルストーンを、ID・内容・Boundary(担当範囲)・Depends(依存タスク)・
   完了条件を持つ具体的なタスクに分解し、`tasks.md` に出力する。分解が粗いと、次の独立
   サブエージェントへの委譲が機能しない。
6. **実装の進め方**: 1タスクずつ「実装者(独立・TDD の RED→GREEN)」「レビューア(Stage 1
   と同じ)」「デバッガー(reject 2回で発動、最大2ラウンド)」のサブエージェント委譲で進める。
   横断的な学びは `tasks.md` の `## Implementation Notes` に積み上げ、後続タスクに引き継ぐ。
   マイルストーンごとにダミーデータで動作確認しユーザーと合意してから次に進む(対話的に自分で
   書きたい場合はこの委譲を省略してよいが、独立レビューは必ず行う)。
7. **統合検証(GO/NO-GO)**: 全タスク完了後、全テストスイートの実行結果・タスク間インター
   フェースの整合・仕様への対応漏れを独立して検証し、`GO`/`NO-GO`/`MANUAL_VERIFY_REQUIRED`
   の3値で判定する。

### 出力

```
research/
    implementation-plan.md  # マイルストーン、優先順位、実験管理方針、リスク
    architecture.md         # ディレクトリ構成、モジュール設計、クラス/API設計、データ構造
    test-plan.md            # テスト一覧(目的・入力・期待値)、CI設計
    tasks.md                # タスク一覧(ID・内容・Boundary・Depends・完了条件)、
                             # Implementation Notes
```

### 品質基準

- 「ベースラインを先に動かす」を原則とする(提案手法の評価はベースラインの再現ができてから)
- 実験の設定・結果が後から追跡できる設計にする(再現性フェーズを楽にする)
- 評価に関わる部分(指標計算・データ分割)のテストは必須とする
- タスクは Boundary を明確にし、担当外のモジュールに手を入れない
- 全タスク完了後は統合検証(GO/NO-GO)を経てから次のステップへ進む

### 実装コードの独立レビュー

この設計書をもとに実際の実験コードを書き始める場合、書いたそばから自己判断だけで正しさを
確認しない。`research-08-evaluation/references/code-review.md` にある、過去の会話コンテキストを
渡さないサブエージェントによる3段階レビュー(設計適合・バグ・境界チェック → ダミーデータ
での妥当性検証(RED→GREEN)→ 実データ適用後の結果妥当性)を、test-plan.md のテスト計画と
合わせて適用する。同じ指摘で2回連続 reject されたらデバッガー役のサブエージェントを挟む。

### 実装の進め方

- 専用ブランチを切って実装し、タスクごとにコミットする。品質(独立レビュー・統合検証)に
  問題がないことをユーザーと合意できてから元のブランチにマージする
- ユーザーに確認を依頼する際、学習曲線や出力サンプルなど図がある場合はテキストの羅列だけで
  済ませず、図を埋め込んだ HTML で提示する(Artifact tool があれば使う)

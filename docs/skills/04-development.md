# Phase 4: Development

アルゴリズム設計を、実装可能な詳細設計書とタスク一覧に落とし込み、それを実際にコードとして
実装するフェーズです。「何をどう作るか決める」(research-07-implementation-plan)と「実際に手を
動かして作る」(research-08-implementation-execution)を分けています。

## research-07-implementation-plan

| 項目 | 内容 |
| --- | --- |
| コマンド | `/research-07-implementation-plan` |
| 使用モデル | Sonnet |
| 目的 | 実装計画・アーキテクチャ・モジュール構成・テスト計画・タスク分解を作成する(コードは書かない) |

### いつ使うか

- 「実装計画を立てたい」「アーキテクチャを設計したい」「実験コードの設計」
  「テスト計画を作りたい」「タスクに分解したい」
- `research-06-algorithm-design` の次のステップとして

### 主な処理

1. **仕様の一貫性チェック**: `algorithm-design.md`・`pseudo-code.md`・`constraints.md`・
   `evaluation-metrics.md` を読み、詳細設計に必要な情報のあいまいさや、仕様書間の矛盾が
   ないか確認する。見つかれば決め打ちせず、`AskUserQuestion` でユーザーに方針を確認する。
2. **全体アーキテクチャ**: ディレクトリ構成、モジュール分割(擬似コードのアルゴリズム単位と
   1:1 対応させる)、設定管理・実験管理方針を設計する。
3. **詳細設計**: 主要クラス・関数のシグネチャ、データ構造、学習ループ・推論パイプラインを設計する。
   提案手法とベースラインを同一の評価コードで測れるインターフェースにする。
4. **テスト計画**: ユニットテスト(数式どおりの動作検証)、形状・数値健全性テスト、統合テスト、
   再現性テスト、過学習サニティチェックを設計する。
5. **CI・実装順序**: MVP → ベースライン再現 → 提案手法フル → アブレーション対応、の順で
   マイルストーンと完了条件を定める。
6. **タスク分解**: 各マイルストーンを、ID・内容・Boundary(担当範囲)・Depends(依存タスク)・
   完了条件を持つ具体的なタスクに分解し、`tasks.md` に出力する。分解が粗いと、
   `research-08-implementation-execution` での独立サブエージェントへの委譲が機能しない。

### 出力

```
research/
    implementation-plan.md  # マイルストーン、優先順位、実験管理方針、リスク
    architecture.md         # ディレクトリ構成、モジュール設計、クラス/API設計、データ構造
    test-plan.md            # テスト一覧(目的・入力・期待値)、CI設計
    tasks.md                # タスク一覧(ID・内容・Boundary・Depends・完了条件)、
                             # Implementation Notes(空)
```

### 品質基準

- 「ベースラインを先に動かす」を原則とする(提案手法の評価はベースラインの再現ができてから)
- 実験の設定・結果が後から追跡できる設計にする(再現性フェーズを楽にする)
- 評価に関わる部分(指標計算・データ分割)のテストは必須とする
- タスクは Boundary・Depends を過不足なく書く。あいまいなタスクは委譲したときに機能しない
- 仕様書間のあいまいさ・矛盾は自分で解釈を決め打ちせず、ユーザーに方針を確認してから設計に反映する

## research-08-implementation-execution

| 項目 | 内容 |
| --- | --- |
| コマンド | `/research-08-implementation-execution` |
| 使用モデル | Sonnet |
| 目的 | `tasks.md` のタスクを実際にコードとして実装し、独立レビュー・統合検証で品質を担保する |

### いつ使うか

- 「実装を進めたい」「タスクを実装したい」「コードを書きたい」「実装を実行したい」
- `research-07-implementation-plan` の次のステップとして

### 主な処理

1. **タスクごとのサブエージェント委譲**: 1タスクずつ「実装者(独立・TDD の RED→GREEN)」
   「レビューア(独立、設計適合・境界チェック)」「デバッガー(reject 2回で発動、最大2ラウンド)」
   に委譲する。過去の会話コンテキストは渡さず、タスク単体の情報だけで完結させる。
2. **タスク完了時の処理**: 横断的な学びは `tasks.md` の `## Implementation Notes` に積み上げ、
   後続タスクに引き継ぐ。専用ブランチにタスク単位でコミットする。
3. **マイルストーンの確認**: マイルストーンごとにダミーデータで動作確認し、図があれば埋め込んだ
   HTML でユーザーに提示して合意を得てから次に進む。
4. **統合検証(GO/NO-GO)**: 全タスク完了後、全テストスイートの実行結果・タスク間インター
   フェースの整合・仕様への対応漏れを独立して検証し、`GO`/`NO-GO`/`MANUAL_VERIFY_REQUIRED`
   の3値で判定する。

対話的に自分で書きたい場合はサブエージェントへの委譲を省略してよいが、独立レビューは必ず行う。

### 出力

```
research/
    tasks.md              # 完了状態・Implementation Notes を更新
    code-review-log.md    # 各タスクのレビュー結果(PASS/FAIL・指摘・対応)
```

実装コード自体は `architecture.md` の設計に従ったソースツリーに出力する(`research/` 配下ではない)。

### 実装コードの独立レビュー

書いたそばから自己判断だけで正しさを確認しない。`references/code-review.md` にある、過去の
会話コンテキストを渡さないサブエージェントによる2段階レビュー(設計適合・境界チェック →
RED→GREEN の妥当性検証)を適用する。同じ指摘で2回連続 reject されたらデバッガー役の
サブエージェントを挟む。

### 実装の進め方

- 専用ブランチを切って実装し、タスクごとにコミットする。品質(独立レビュー・統合検証)に
  問題がないことをユーザーと合意できてから元のブランチにマージする
- ユーザーに確認を依頼する際、学習曲線や出力サンプルなど図がある場合はテキストの羅列だけで
  済ませず、図を埋め込んだ HTML で提示する(Artifact tool があれば使う)

### 品質基準(両スキル共通)

- 全タスク完了後は統合検証(GO/NO-GO)を経てから次のステップへ進む。個別タスクが全て
  レビューを通過していても、タスク間のインターフェースが噛み合っているとは限らない
- 「直った」「動いた」は記憶や以前の実行結果ではなく、その場で再実行した新しい証拠で宣言する

# Phase 3: Design

研究仮説を解く新手法を考案し、知財面を確認したうえでアルゴリズムとして定式化するフェーズです。

## research-04-method-design

| 項目 | 内容 |
| --- | --- |
| コマンド | `/research-04-method-design` |
| 使用モデル | Opus |
| 目的 | 複数の新手法案を考案・比較し、採用案を決定する |

### いつ使うか

- 「手法を考えたい」「アプローチを検討したい」「解決策のアイデアがほしい」
  「どの方法を採用するか決めたい」
- `research-03-gap-analysis` の次のステップとして

### 主な処理

1. **解決案の発散**: 性質の異なる手法案を最低 3 つ生成する(既存手法の改良型、異分野からの転用型、
   組み合わせ型など)。
2. **体系的な比較評価**: 新規性・実現可能性・計算コスト・制約適合性・期待効果・リスクの軸で
   スコア表を作る。新規性の主張は文献と突き合わせる。
3. **採用案の決定**: 推奨案をユーザーに確認して決定する。不採用案とその理由も記録する
   (後のアブレーション設計・査読対応で再利用する)。

### 出力

```
research/
    method-proposals.md   # 全手法案の詳細、比較スコア表、不採用理由
    selected-method.md    # 採用案の詳細、新規性の言明、段階的な実装計画、将来拡張案
```

## research-05-patent-search

| 項目 | 内容 |
| --- | --- |
| コマンド | `/research-05-patent-search` |
| 使用モデル | Sonnet |
| 目的 | 提案手法に対する特許・知財調査を行い、新規性・侵害リスクを確認する |

> このスキルの調査結果は参考情報であり、法的助言ではありません。事業判断・出願判断の前には
> 弁理士等の専門家に確認してください。

### いつ使うか

- 「特許を調べたい」「知財調査」「特許侵害リスクを確認したい」
- `research-04-method-design` の次のステップとして(学術研究のみで知財が関係しない場合は
  スキップして `research-06-algorithm-design` に進んでもよい)

### 主な処理

1. Google Patents・J-PlatPat・USPTO/Espacenet で日英キーワードによる検索を行う。
2. 類似特許のクレームと自手法の構成要素を対比表にする。
3. 新規性評価と侵害リスク評価(高/中/低 + 理由)をまとめ、リスクが高い場合は差別化案を提示する。

### 出力

```
research/
    patent-review.md      # 調査方法、新規性評価、侵害リスク評価、免責事項
    patent-comparison.md  # 類似特許の詳細、クレーム対比表、差別化案
```

## research-06-algorithm-design

| 項目 | 内容 |
| --- | --- |
| コマンド | `/research-06-algorithm-design` |
| 使用モデル | Opus |
| 目的 | 提案手法を数式・アルゴリズム・疑似コード・フローチャートへ落とし込む |

### いつ使うか

- 「アルゴリズムを設計したい」「数式に落としたい」「擬似コードを書きたい」「計算量を解析したい」
- `research-05-patent-search`(または `research-04-method-design`)の次のステップとして

### 主な処理

1. **数学的定式化**: 記法表を最初に作り、問題設定と提案手法を数式で表現する。
2. **アルゴリズム記述**: 学習・推論の擬似コードを書く。停止条件・エッジケースを明記する。
3. **フロー**: Mermaid で全体フローチャート・データフロー図を作成する。
4. **計算量解析**: 時間・空間計算量を導出過程込みで示し、ベースラインと比較する。

### 出力

```
research/
    algorithm-design.md   # 記法表、問題設定、定式化、計算量解析
    pseudo-code.md        # 学習・推論の擬似コード
    flowchart.md          # Mermaid フローチャート・データフロー図
```

### 品質基準(3スキル共通)

- `research-04` の構成要素分解 → `research-06` の擬似コードのモジュール分割が対応するように設計する
  (アブレーション実験で個別に外せる単位にする)
- 新規性・侵害リスクの主張は検索で確認した一次情報に基づく
- 「実装者がこの文書だけで実装に着手できるか」を完成基準とする

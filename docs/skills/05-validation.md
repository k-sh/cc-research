# Phase 5: Validation

提案手法の性能を統計的に信頼できる形で評価し、寄与要因を分析し、第三者が再現できる状態に
整えるフェーズです。

## research-09-evaluation

| 項目 | 内容 |
| --- | --- |
| コマンド | `/research-09-evaluation` |
| 使用モデル | Sonnet |
| 目的 | 評価実験を設計・実施し、ベースライン比較・統計評価・考察を行う |

### いつ使うか

- 「実験したい」「評価したい」「ベースラインと比較したい」「統計解析したい」
- `research-07-implementation-plan` の次のステップとして

### 主な処理

1. **実験計画の事前登録**: 結果を見る前に、検証する仮説・データセット・ベースライン・評価指標・
   乱数シード(最低3つ)・統計検定の方法を確定させる。
2. **実施**: 極小データでのドライラン → 本実験。長時間ジョブはバックグラウンド実行で監視する。
   失敗した実行も原因とともに記録する。
3. **統計解析と可視化**: 平均±標準偏差、統計検定の p 値と効果量、学習曲線・比較・トレードオフの
   グラフを作成する。
4. **考察と限界**: 仮説の支持/棄却を明言し、予想と違った結果も正直に分析する。

実験コードは自己レビューだけで進めない。過去の会話コンテキストを渡さない独立したサブエージェント
による3段階レビュー(実装中の設計適合・バグ・境界チェック → 完成時のダミーデータでの妥当性検証
(RED→GREEN)→ 実データ適用後の結果妥当性の検証)を必須の工程とする。Stage 1 では設計書の
項目とコードの対応関係を1つずつ確認し、`git diff`・TODO の grep・テスト実行も行う。Stage 2 では
判定ロジック実装前に「期待通りにならない」ことを確認した RED-phase の証拠を残す。Stage 3 では
「うまくいかなかった」結果をまず実装バグと疑って最も単純な(手計算・目視で正解が分かる規模の)
ケースで再評価したうえで、統計指標だけでなく代表的な出力サンプル(画像・生成物など)を独立
レビューさせ、目的を実際に達成しているように見えるかも確認する。対象は結果を左右するコアロジック
の変更時に限り、レビュー用サブエージェントには実装者と同格以上のモデル(結果解釈を扱う Stage 3
は既定で opus)を明示的に指定する。同じ指摘で2回連続 reject されたら、闇雲な再修正の前に
「デバッガー」役の独立サブエージェント(root cause 調査、最大2ラウンド)を挟む。それでも
解消しない場合はユーザーに判断を仰いで打ち切る。「直った」「うまくいった」はその場で再実行した
新しい証拠でのみ宣言する。手順・プロンプト例は `research-09-evaluation/references/code-review.md`
を参照。結果をユーザーに確認してもらう際は、グラフ・出力サンプルを埋め込んだ HTML でまとめて
提示する(Artifact tool があれば使う)。

### 出力

```
research/
    evaluation-report.md    # 実験計画(事前登録)、結果サマリ、統計解析、考察、限界
    experiment-results.md   # 生データ(全実行の条件・シード・結果)
    figures/                # グラフ
    figures/samples/        # 代表的な出力サンプル(独立レビューでの目視評価用)
```

## research-10-ablation-study

| 項目 | 内容 |
| --- | --- |
| コマンド | `/research-10-ablation-study` |
| 使用モデル | Sonnet |
| 目的 | アブレーション・感度分析・失敗分析により各要素の寄与を評価する |

### いつ使うか

- 「アブレーションしたい」「どの要素が効いているか調べたい」「感度分析」「失敗分析」
- `research-09-evaluation` の次のステップとして

### 主な処理

1. **アブレーション設計**: 構成要素ごとに「外す/置き換える」実験を列挙し、優先順位を付ける。
2. **実施**: 評価実験と同じプロトコル(同シード集合)でフル手法・各アブレーション条件・
   ベースラインを比較する。
3. **分析**: 寄与率分析、パラメータ感度分析、学習曲線・推論時間・メモリの効率分析、
   失敗事例のエラー分析、ボトルネック分析を行う。

research-09-evaluation と同様、実験コードは `references/code-review.md` の3段階の独立レビュー
(設計適合・バグ・境界チェック → ダミーデータでの妥当性検証(RED→GREEN) → 実データ適用後の
結果妥当性)を通してから結論を確定する。「効いていない」結果はまず実装バグを疑い、統計指標
だけでなく代表的な出力サンプルも独立レビューさせる。同じ指摘で2回連続 reject されたらデバッガー
役のサブエージェントを挟む。結果をユーザーに確認してもらう際は、グラフ・出力サンプルを
埋め込んだ HTML でまとめて提示する。

### 出力

```
research/
    ablation-study.md        # アブレーション条件表、結果、寄与率分析、エラー分析
    sensitivity-analysis.md  # ハイパーパラメータ感度、効率分析、頑健性の総括
    figures/samples/         # 代表的な出力サンプル(独立レビューでの目視評価用)
```

## research-11-reproducibility

| 項目 | 内容 |
| --- | --- |
| コマンド | `/research-11-reproducibility` |
| 使用モデル | Haiku |
| 目的 | 実験環境・依存関係・データ・設定を整理し、第三者が再現できる状態にする |

### いつ使うか

- 「再現性を確保したい」「環境をまとめたい」「実験手順書を作りたい」
- `research-10-ablation-study` の次のステップとして

### 主な処理

1. **環境の棚卸し**: `uname`・`pip freeze`・`nvidia-smi` 等の実コマンド出力から環境情報を取得する
   (記憶や推測では書かない)。
2. **実験条件の棚卸し**: 乱数シード、学習・推論設定、データセットの入手方法とライセンスを整理する。
3. **再現手順書**: クリーンな環境から主要結果を再現するまでの手順を書き、可能な限り
   1 コマンドで再現できるスクリプトを用意する。
4. **Docker 方針とチェックリスト**: Dockerfile 作成方針と再現性チェックリストをまとめる
   (`code-review-log.md` の有無で実験コードの独立レビューが実施済みか、`tasks.md` で統合検証
   (GO/NO-GO)が `GO` だったかも確認項目に含める)。

### 出力

```
research/
    reproducibility.md    # 総括とチェックリスト、Docker方針
    environment.md         # 環境情報(コマンド出力ベース)、依存バージョン固定
    experiment-guide.md    # 再現手順書
```

### 品質基準(3スキル共通)

- シード 1 つの結果で結論を出さない
- 環境情報は必ず実コマンドの出力から書く
- 再現できない要素(独自データ・巨大計算資源)は隠さず明記する

# Phase 2: Survey

既存研究を調査し、研究として解くべきギャップと仮説を特定するフェーズです。

## research-02-literature-review

| 項目 | 内容 |
| --- | --- |
| コマンド | `/research-02-literature-review [調査トピック]` |
| 使用モデル | Sonnet |
| 目的 | Google Scholar・arXiv・GitHub・企業ブログ・公式ドキュメントから既存研究とベースラインを整理する |

### いつ使うか

- 「先行研究を調べたい」「文献調査」「サーベイ」「関連研究をまとめたい」「ベースラインを探したい」
- `research-01-planning` の次のステップとして

### 主な処理

1. **検索**: `research-plan.md` の調査キーワード(日英)を使い、arXiv・Google Scholar・GitHub・
   企業ブログ・公式ドキュメントを調査する。最新研究と被引用数の多い基礎研究の両方を押さえる。
   検索対象が多い場合は Haiku サブエージェントに並列で調査を分担させ、コストを抑えてよい。
   ただしサブエージェントには成果物ファイルを書き込ませず、調査結果は回答テキストとして
   回収する(ファイル書き込みは本体が一括して行い、並行書き込みによる競合を避ける)。
2. **引用の実在検証**: 記憶から論文を書き出さず、検索・実ページで存在を確認できたものだけを
   一覧に載せる(URL/DOI 必須)。
3. **整理**: 関連研究を 3〜6 カテゴリに分類し、比較表とベースライン候補(再現可能性・計算コスト)
   をまとめる。

### 出力

```
research/
    literature-review.md   # 調査方法、分類体系、時系列の流れ、主要な発見
    related-works.md       # 文献ごとの詳細カード + 分類別比較表
    baseline-summary.md    # ベースライン候補(再現可能性・計算コスト・推奨順位)
```

## research-03-gap-analysis

| 項目 | 内容 |
| --- | --- |
| コマンド | `/research-03-gap-analysis` |
| 使用モデル | Opus |
| 目的 | 研究ギャップ・未解決課題を抽出し、検証可能な研究仮説を立案する |

### いつ使うか

- 「研究ギャップを見つけたい」「未解決課題を整理したい」「仮説を立てたい」「何を解くべきか決めたい」
- `research-02-literature-review` の次のステップとして

### 主な処理

1. **課題の抽出**: 各論文の明示的な限界(Limitation/Future Work)と、比較表から読み取れる
   暗黙のギャップ(未探索の条件・組み合わせ、実運用との乖離)の両方を集める。
2. **構造化**: 技術課題・実運用課題・評価課題に分類し、複数論文に共通する課題を強調する。
3. **ギャップの特定**: 「既存研究は A を達成しているが B の条件下では C ができていない」形式で
   ギャップを言明し、差別化ポイント候補を挙げる。
4. **仮説の立案と優先順位付け**: 各仮説について根拠・検証方法・棄却条件・期待インパクトと
   実現可能性を整理し、最優先の仮説をユーザーと確認する。

### 出力

```
research/
    research-gap.md      # 研究ギャップの言明、差別化ポイント
    technical-issues.md  # 技術課題・実運用課題・評価課題の分類
    hypotheses.md        # 検証可能な仮説一覧(根拠・検証方法・棄却条件・優先順位)
```

### 品質基準(両スキル共通)

- 「まだ誰もやっていない」だけでなく「やる価値があるのに難しくてできていない」ことを示す
- 仮説は反証可能な形式で書く(「〜を改善する」ではなく「条件 X で指標 Y が Z 以上変化する」)
- 文献に基づく主張には必ず出典(`related-works.md` のエントリ)を付ける

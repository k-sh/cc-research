# 評価実験(Evaluation)

提案手法の性能を統計的に信頼できる形で評価し、レポートにまとめる。

## 前提

開始時に `research/STATUS.md`・`research/evaluation-metrics.md`・`research/hypotheses.md`・`research/baseline-summary.md`・`research/implementation-plan.md`・`research/tasks.md`(あれば)を読む。
`tasks.md` があれば `## Implementation Notes` に既知のハマりどころが無いか確認する。
実装コードの有無と実行環境を確認する。

## 進め方

### 1. 実験計画の事前登録(実行前に必ず書く)

**結果を見る前に**実験計画を `research/evaluation-report.md` の冒頭に確定させる。後から都合よく評価条件を変えないための予防措置:

- 検証する仮説(hypotheses.md への参照)と判定基準
- データセット(分割方法、前処理、リーク防止の確認)
- ベースライン(バージョン・設定を固定)
- 評価指標と測定方法(evaluation-metrics.md と一致させる)
- **乱数シード: 最低 3 つ**(可能なら 5)で複数回実行
- ハイパーパラメータ探索の範囲と方法(提案手法とベースラインに同等の探索予算を割く)
- 統計検定の選択(対応の有無・正規性を考慮)と有意水準

### 2. 実験の実施

評価用のコード(指標計算・統計処理など)自体の設計適合レビューは、コアな実装であれば
`research-08-implementation-execution` が既に済ませている。ここで新たに独立ソースレビューを
重ねる必要はない。ただし、動けば正しいと思い込むのは禁物: 極小データでのドライランで配線を
確認してから本実験を回す(自分で行う軽い自己チェックで十分、独立レビューまでは不要)。

- 長時間ジョブは Bash の `run_in_background` で実行し、途中経過を監視する
- **複数条件・複数シードを並行実行する場合、各ジョブの出力先は条件ごとに別ファイル
  (例: `research/logs/seed{N}-{condition}.log`)にする。** 複数のバックグラウンドジョブが
  `research/experiment-results.md` に同時に追記すると、書き込みが競合して内容が消失・破損する。
  `research/experiment-results.md` への集約は、各ジョブの完了を確認したうえで自分(本体)が
  一つずつ読み取って書き足す形にする(並行書き込みをさせない)
- 生の結果(全シード・全条件)を `research/experiment-results.md` に追記していく。失敗した実行も原因とともに記録する

### 3. 統計解析と可視化

- 各条件: 平均 ± 標準偏差(または信頼区間)を報告
- ベースラインとの差: 統計検定の p 値 + **効果量**(差の大きさ)。「有意だが微小」を見分ける
- グラフ作成(matplotlib 等): 学習曲線、条件別比較、指標のトレードオフ図。軸ラベル・単位・凡例を必ず付ける
- 図は `research/figures/` に保存(論文・スライドで再利用する)

### 4. 考察と限界

- 仮説は支持されたか棄却されたかを明言する(グレーなら「判定不能」とその理由)
- 予想と違った結果の原因分析(悪い結果を隠さない — アブレーションと査読対応の材料になる)
- 限界の整理: 評価できていない条件、一般化の範囲、脅威(threats to validity)
- **結論を確定する前に `references/code-review.md` の結果の妥当性レビューを通す。** 「うまく
  いった」結果もコードのバグ・データリークで説明できないか、「うまくいかなかった」結果も
  実装のバグではなく手法自体の限界と言い切れるかを、独立したサブエージェントに検証させる。
  **コードの問題が疑われれば前工程(実装)に戻って直し、再実行してからこのレビューをやり直す**
  (実装を直した勢いで結論を確定しない)。手法自体の限界であればここに正直に反映する
- **「うまくいかなかった」場合はまず実装バグを疑う。** 新手法が「本当に効果がない」より
  「どこかにバグがある」方が圧倒的に多い。手法の限界と結論する前に、最も単純なケース
  (差が最も大きく出るケースではなく、手計算・目視で正解が分かるくらい小さい入力)で
  改めて妥当な結果が出るかを再評価する
- **統計だけで「うまくいった/いかなかった」を判断しない。** 代表的な出力サンプル(画像・
  生成テキスト・予測例など)を `research/figures/samples/` に保存し、独立レビューで統計を
  見せる前にまず出力そのものを見て目的を達成しているように見えるか判定させる
  (詳細は `references/code-review.md` を参照)

### 5. 出力

```
research/
    evaluation-report.md    # 実験計画(事前登録)、結果サマリ表、統計解析、考察、限界
    experiment-results.md   # 生データ: 全実行の条件・シード・結果・ログ参照
    figures/                # グラフ
    figures/samples/        # 代表的な出力サンプル(独立レビューでの目視評価用)
```

### 6. 完了処理

- `research/STATUS.md` を更新(仮説の判定結果を記録)
- 結果をユーザーに確認してもらう際は、グラフ・出力サンプルを埋め込んだ HTML でまとめて提示する
  (Artifact tool が使える環境ならそれで公開し、無ければファイルに保存してパスを案内する)。
  ファイルパスや数値の羅列だけでは、図の内容が伝わらず確認の質が落ちる
- 次のステップ `/research-10-ablation-study` を案内

## 品質基準

- 結果の妥当性レビュー(`references/code-review.md`)は必ず飛ばさない。ソースコード自体の
  独立レビューは `research-08-implementation-execution` が担うため、ここで重複させない
- 「うまくいかなかった」結果を見たら、まず実装バグを疑う。最も単純なケースで妥当な結果が
  出るかを確認してから、手法自体の限界かどうかを判断する。同じ問題で繰り返し行き詰まる場合は、
  実装ではなく手法の論理自体を疑い、method-design/algorithm-design への差し戻しを検討する
- 統計指標だけで成功・失敗を判定しない。代表的な出力サンプルを実際に見て(独立レビューさせて)
  目的を達成しているように見えるかを確認する
- シード 1 つの結果で結論を出さない
- ベースラインの数値は自環境での再実行値を第一とし、論文からの引用値を使う場合はその旨を明記する
- 表の数値は experiment-results.md の生データから機械的に導出できること(手作業の転記ミスを防ぐため集計スクリプトを書く)

# 実験結果の独立レビュー

研究実験のコードは「動いた」ことと「意図通りに動いた」ことが一致しない場合が多い
(指標の定義ミス、データリーク、シード未固定、統計検定の選択ミスなどは実行時エラーにならない)。
ソースコード自体の設計適合レビューは `research-08-implementation-execution` が担うため、ここでは
それを重複させない。その代わり、**実データで結果が出た後**、過去の会話コンテキストを渡さない
独立したサブエージェントに、統計上の結論・生の出力の両面から結果の妥当性をレビューさせる。
結果に問題がある(コードのバグが疑われる)場合は、前工程(実装)に戻って直し、再実行してから
このレビューをやり直す。

**目次**: [共通ルール](#共通ルール)(実行方法・出力フォーマット・指摘への対応・デバッガー・
完了宣言前のフレッシュな証拠)/ [結果の妥当性レビュー](#結果の妥当性レビュー)(統計上の結論・
出力そのものの目視評価)/ [記録](#記録)

## 共通ルール

### いつ行うか

実データで実験を実行し、結果が出た後、`evaluation-report.md` の結論を確定する前に必ず1回行う。
瑣末な文言修正のためにレポートを書き直すたびに毎回回す必要はないが、結果の解釈・結論に関わる
変更をした場合は都度やり直す。

### レビューの実行方法

- レビューは Agent tool でサブエージェント(`general-purpose` 想定)を起動して行う。
- サブエージェントには、それまでの会話・設計判断の経緯を渡さない。渡すのは
  「参照すべき仕様書のパス」「結果データ・ドラフトのパス」「レビューさせたい観点」のみ。
- **モデル**: 判断の難度が高い(統計的な結論の妥当性・生の出力の質を横断的に評価する)ため、
  Agent tool の `model` パラメータで既定 `opus` を明示的に指定する。
- プロンプトに「これで合っているはず」「一応確認して」のような期待値・誘導を書かない。
  独立した第三者として虚心にジャッジさせる。

### 出力フォーマット(構造化)

サブエージェントには自由記述ではなく、次の形式で厳密に返させる。曖昧な文章判定は
親エージェント側での見落とし・誤読を招くため、機械的に次の一手を判断できる形にする
(下記の各プロンプト例には、この指定をそのまま含めてある):

```
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容(該当ファイル・箇所・何が期待と違うか)
(指摘がない場合は「指摘: なし」と明記)
```

### 指摘への対応

- 重大度「高」(結論を覆しうるもの)は必ず解消してから次に進む。「中」「低」は対応するか、
  対応せず記録だけして先に進むかを自分で判断してよい。
- **レビュアーの判定は絶対ではない。** 指摘の根拠が薄い、または明確に反証できる場合は、
  盲目的に修正せず、指摘とその反証をユーザーに提示して判断を仰ぐ。
- **同じ指摘で2回連続 reject されたら、闇雲に3回目の修正を試みない。** 代わりに
  「デバッガー」役の独立したサブエージェントを起動する(過去の会話コンテキストを渡さない。
  原因調査に必要なら WebSearch/WebFetch の使用も許可する)。デバッガーには、指摘内容・
  これまで試した修正・関連コードを渡し、次を返させる:
  - `ROOT_CAUSE`: 何が根本原因か
  - `CATEGORY`: 「うまくいかなかった」場合の原因分析の順番(下記)に対応させ、
    実装バグ/テストデータの不整合(座標系・単位・時間軸など)/環境要因(他プロセス・熱
    スロットリングなど)/先行研究に対する認識齟齬/**手法自体の論理の誤り**、のいずれかで返す
  - `FIX_PLAN`: 次に何をどう直すべきかの具体的な手順
  この fix plan に沿って修正し、レビューをやり直す。**デバッガーのラウンドは最大2回まで。**
  2回のデバッグでも同じ指摘が解消しない場合はレビューを打ち切り、ユーザーに状況(指摘内容・
  試した修正・デバッガーの調査結果)を報告して判断を仰ぐ。無限にレビュー→修正を繰り返さない。
  特に `CATEGORY` が「手法自体の論理の誤り」に近いと判断された場合は、実装を直し続けるのではなく
  `research-04-method-design`・`research-06-algorithm-design` への差し戻しをユーザーに提案する。

### 完了宣言の前に(フレッシュな証拠)

「直った」「うまくいった」「PASS した」とは、記憶や以前の実行結果だけで言わない。必ず
その場で再実行し、新しく得られた証拠(出力・ログ・テスト結果)を確認してから宣言する。
修正を加えた直後に「これで直ったはず」で終わらせず、実際に再実行して確認できるまでは
未検証(NOT_VERIFIED)として扱い、ユーザーにも「未検証」と「検証済み」を混同させない。

## 結果の妥当性レビュー

**いつ**: 実データで実験を実行し、結果が出た後、`evaluation-report.md` の結論を確定する前。

**渡す情報**: `research/hypotheses.md`、事前登録した実験計画、`research/experiment-results.md`
(生データ)、`evaluation-report.md` のドラフト(結果サマリと考察)。

**観点**:
- 「うまくいった」場合: その改善が本物の効果によるものか、それともコードのバグ・データリーク・
  比較条件の不公平さ・シード数不足など別の要因で説明できてしまわないか。
- 「うまくいかなかった」場合: **原因分析は次の順番で行い、手法そのものを疑うのは最後の手段と
  する。** 先行研究や妥当性の高い仮説に基づいて研究している以上、提案手法が多少想定と異なる
  (悪化する)ことはあっても、「全く使い物にならない」レベルの結果になる原因が手法そのものに
  あることは通常考えにくい。次の順で1つずつ消していく:
  1. **実装を疑う。** 実装バグ、使用ライブラリ・APIの使い方の認識齟齬がないか。**最も単純な
     ケース**(差が最大に出るように設計したケースではなく、手計算・目視で正解が分かるくらい
     小さく作った入力。test-plan.md のユニットテスト・過学習サニティチェックと同じ規模感)で
     改めて実行し、期待通りの結果が出るか確認する
  2. **テストデータを疑う。** 特に複数のデータソースを組み合わせている場合、一部だけ時間軸・
     座標系・単位が異なっている、といったデータ由来の不整合がないか確認する
  3. **環境を疑う。** 特に処理速度・メモリ使用量など効率面の指標が想定と違う場合、同一サーバ上の
     他プロセスや熱スロットリングなど、コード自体とは無関係な環境要因がないか確認する
     (専有環境・複数回計測での再現性を確認する)
  4. **方法に対する認識齟齬を疑う。** 参考にしている先行研究(related-works.md・元論文)を
     読み直し、自分の理解・実装が論文の主張・手順と食い違っていないか確認する
  5. **方法そのものを疑う。** ここまでの4つを消してもなお説明がつかない場合に初めて、
     提案手法自体の限界と判断する

  この順番を飛ばして「手法の限界」と結論しない。最も単純なケースですら期待通りにならない
  場合はほぼ確実に1〜4のいずれかが原因であり、複雑な実データでの「効果なし」をそのまま手法の
  結論として採用しない
- シード数・統計検定・効果量の報告が、その結論を支えるのに十分な強さか。
- **多重比較の補正。** 複数指標・複数条件・複数シードにわたって有意差検定を行っている場合、
  Bonferroni 補正等を考慮しているか、それとも単一の有意水準を使い回して第一種の過誤
  (偶然の有意差を実際の効果と誤認するリスク)を膨らませていないか。
- 好都合な結果だけを見て、都合の悪い実行結果(失敗ログなど)を無視していないか。

**プロンプト例**:

```
あなたはこの実験の実施には関与していない第三者のレビュアーです。
以下の仮説・実験計画・生の実験結果・導かれた結論を読み、結論の妥当性を検証してください。

仮説と実験計画: {hypotheses.md, evaluation-report.md 冒頭の事前登録部分}
生データ: {experiment-results.md}
導かれた結論: {evaluation-report.md の考察部分}

「うまくいった」結論であれば、それがコードのバグ・データリーク・比較条件の不公平さなど
別の要因で説明できてしまわないか検討してください。
「うまくいかなかった」結論であれば、次の順で原因分析が行われているか確認してください:
(1) 実装バグ・ライブラリ/APIの誤用、(2) テストデータの不整合(座標系・単位・時間軸など)、
(3) 環境要因(他プロセス・熱スロットリングなど、特に速度・メモリ指標で)、(4) 先行研究に
対する認識齟齬、(5) 手法そのものの限界。この順番のうち確認が飛ばされているステップが
あれば指摘してください。特に、最も単純なケース(差が最も大きく出るケースではなく、手計算・
目視で正解が分かるくらい小さい入力)での再評価は(1)の一部として必須です。
結論を支える証拠(シード数・検定・効果量)が十分かも評価してください。
複数の指標・条件・シードにわたって検定を行っている場合、多重比較の補正が必要かどうか、
必要なら考慮されているかも確認してください。

出力は次の形式で厳密に返してください:
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容(該当ファイル・箇所・何が期待と違うか)
(指摘がない場合は「指摘: なし」と明記)
```

### 統計だけでなく、出力そのものを見る

統計指標(平均・有意差・効果量)が良くても、それだけでは「本当に目的を達成しているか」を
判断しきれないことが多い。指標上は改善していても、生成物を実際に見るとモード崩壊・不自然な
繰り返し・意味不明な出力・テンプレートの丸暗記といった兆候が見えることがある。逆に指標が
ぱっとしなくても、出力自体は目的を捉えていることもある(指標の選び方が的外れな兆候)。
統計処理を経由しない、生の出力そのものに対する評価をここに追加する。

- 実行主体(自分)は、条件ごとに**代表的な出力サンプル**(良さそうな例・悪そうな例・指標が
  中央値付近の例を含む複数点)を用意し、`research/figures/samples/` 等、独立レビュアーが
  直接見て判断できる形式(画像、生成テキスト、予測と正解の対比など)で保存する
- 独立したサブエージェントに、**まず統計結果を見せずに**、目的(`research/objective.md`・
  `research/hypotheses.md`)とサンプル出力のパスだけを渡し、「これは目的を達成しているように
  見えるか」を判定させる。画像等は説明文で済まさず、Read ツールで直接開いて見られるように
  パスをそのまま渡す
- そのうえで統計結果(experiment-results.md・導かれた結論)も見せ、**見た目の判断と統計上の
  結論が一致するか**を確認させる。一致しない場合(統計は良いのに出力は悪く見える、その逆)は、
  指標の定義・集計方法・サンプルの選び方に問題がないかを疑う
- 出力が画像・テキストのような直接見られる artifact を持たないタスク(スカラー値のみ等)の
  場合は、この評価は該当なしとしてよい。ただしその場合も、予測と正解の対比表など何らかの
  「生の結果を直接見る」手段が本当に無いかを一度検討してから省略する

**プロンプト例(出力そのものの評価)**:

```
あなたはこの実験の実施には関与していない第三者のレビュアーです。
まだ統計結果は見せません。以下の目的と、実際に生成された出力サンプルを見て、
この出力が目的を達成しているように見えるかを判断してください。

目的: {objective.md, hypotheses.md のパス}
出力サンプル: {良さそう/悪そう/中央値付近の出力サンプルのパス一覧}

判断してください:
1. これらの出力は、目的に照らして「うまくいっている」ように見えるか
2. 見た目の質が疑わしい兆候(モード崩壊、不自然な繰り返し、意味不明な出力、テンプレートの
   丸暗記、明らかな失敗パターンなど)はないか

出力は次の形式で厳密に返してください:
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容
(指摘がない場合は「指摘: なし」と明記)
```

上記の判定を得たら、続けて統計結果(experiment-results.md・導かれた結論)を渡し、
「見た目の判断と統計上の結論は一致するか」を追加で判定させる。一致しない場合は重大度「高」の
指摘として扱う。

### 指摘が出た場合

「うまくいかなかった」結果の原因分析の順番(実装 → テストデータ → 環境 → 先行研究の認識齟齬 →
手法そのもの)に沿って、判明した原因ごとに対応する:

1. **実装バグ・ライブラリ/APIの誤用 → 前工程(実装)に戻って直す。** 対象コードが
   `research-08-implementation-execution` で扱ったコアな実装(architecture.md のモジュール)
   であれば、そちらの独立レビュー(`research-08-implementation-execution/references/code-review.md`)
   を使って直す。このスキル自身が書いた評価用のコード(指標計算・統計処理など)であれば、
   その場で直して**再実行し、この結果の妥当性レビューをやり直す**
2. **テストデータの不整合(座標系・単位・時間軸など) → データの取得・前処理コードを直す。**
   複数データソースを組み合わせている箇所を優先的に洗い出す
3. **環境要因(他プロセス・熱スロットリングなど) → 環境を切り分けて再計測する。** 専有環境・
   複数回計測で再現するかを確認してから結果を確定する
4. **先行研究に対する認識齟齬 → 元論文・related-works.md を読み直し、理解を修正した上で
   実装または設計を見直す。** 修正が設計レベルに及ぶ場合は `research-06-algorithm-design` に
   差し戻す
5. **手法そのものの限界 → `evaluation-report.md` の考察・限界セクションに正直に反映する**
   (悪い結果を隠さないという `research-09-evaluation` の品質基準そのもの)。ここに至るのは
   1〜4を全て確認・棄却した後のみ

いずれの場合も、修正後は必ず実データで再実行してから結論を確定する(修正しただけで
「直ったはず」としない)。出力そのものに疑わしい兆候があるのに統計だけ見て「成功」と
判断していた場合も、上記と同じ順で原因(指標の選択・サンプルの偏り・データの問題)を
調査し、統計だけで結論を確定しない。

## 記録

レビュー結果は、判定(PASS/FAIL)・指摘の要約・対応を1エントリだけ短く
`research/code-review-log.md` に残す(仕様書ではないので詳細な議事録は不要)。指摘なしで
一発 PASS した場合も「PASS(指摘なし)」の1行は残す。デバッガーが介入した場合は
`ROOT_CAUSE`/`CATEGORY`/`FIX_PLAN` の要約も残す。

他のタスクにも関わる横断的な学び(例:「このライブラリのこの関数は破壊的変更が入っている」
「この前処理は特定の条件で NaN を出す」)が見つかった場合は、`code-review-log.md` だけに
留めず、`research/tasks.md` の `## Implementation Notes` にも転記する
(`research-08-implementation-execution` 参照)。後続タスクの実装者はここを読んでから着手する
ため、同じ落とし穴を踏まずに済む。

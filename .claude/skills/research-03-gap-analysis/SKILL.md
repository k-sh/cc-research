# ギャップ分析と仮説立案

文献調査の結果から「まだ解かれていない、解く価値のある課題」を特定し、検証可能な研究仮説に落とし込む。

## 前提

開始時に `research/STATUS.md`・`research/literature-review.md`・`research/related-works.md`・`research/baseline-summary.md`・`research/objective.md`・`research/constraints.md` を読む。
文献調査が無い場合は、ユーザーに既知の先行研究状況を質問するか、先に `/research-02-literature-review` の実行を提案する。

## 進め方

### 1. 課題の抽出

2 つの経路で未解決課題を集める:

- **明示的な限界**: 各論文の Limitation / Future Work に書かれていること
- **暗黙のギャップ**: 論文には書かれていないが比較表から読み取れること
  - どの手法も評価していない条件・データ・スケール
  - 手法カテゴリ間の未探索の組み合わせ
  - 研究設定と実運用条件の乖離(レイテンシ、データ品質、運用コスト)

### 2. 課題の構造化

抽出した課題を分類して `research/technical-issues.md` に整理する:

- **技術課題**: アルゴリズム・モデル・計算量の限界
- **実運用課題**: デプロイ・スケール・保守・コストの問題
- **評価課題**: 既存ベンチマーク・指標の不備

複数の論文に共通する課題は「分野の共通課題」として強調する。

### 3. 中核ベースラインの再現検証(任意・推奨)

文献調査だけでは、論文に書かれていない実装上の落とし穴やハイパーパラメータ依存、
実は再現しない主張を見逃すことがある。特定した課題・ギャップ候補が、ある先行研究の
「主張されている結果」を前提にしている場合、その主張を実装ベースで検証してからギャップを
確定させる方が、根拠のないギャップを追いかけるリスクを避けられる。

- **対象を絞る**: 全ての文献を検証しない。`baseline-summary.md` から、ギャップの言明の
  前提になっている中核的な論文に絞る(目安 1〜2 本)。「この論文の主張が崩れたら、想定している
  ギャップ自体が成立しなくなる」ものを選ぶ。実行コストを見積もり、優先順位をユーザーに確認する
  (`research-09-ablation-study` の実験選定と同じ考え方。緊急度・予算次第でこの工程自体を
  スキップしてよい)
- **最小構成で再現する**: 公式実装があればそれを動かして主張を確認するだけでよい。ない場合は、
  論文の記述・擬似コードからコア部分のみ・小規模データで最小構成の再実装をする。この段階の
  目的は「論文の主張の骨格が成り立つか」の確認であり、フル再現や性能の追求は不要
- **独立レビューを軽量版として適用する**: ここで書くコードも「結果に影響する実験コード」に
  変わりない。過去の会話コンテキストを渡さない独立したサブエージェントにレビューさせるという
  考え方・共通ルール(モデル指定、構造化出力、ループ打ち切り基準)は
  `research-08-evaluation/references/code-review.md` の「共通ルール」を流用する。ただし
  この段階では `implementation-plan.md` 等の設計書がまだ無いため、Stage 1 で渡す仕様書は
  元論文の記述・擬似コードに読み替える
- **検証内容**: 論文が主張する傾向・大小関係が再現できるか。再現できない場合は、その理由が
  (a) ハイパーパラメータ・データ・評価条件の違いによるものか、(b) 主張自体が誇張・過小評価
  されている可能性があるのかを切り分ける
- 結果を `research/baseline-validation.md` に記録する: 検証対象、再現方法、再現できたか、
  差異とその原因、ギャップ・仮説への示唆

再現できなかった場合、それ自体が貴重な発見になりうる(先行研究の主張を割り引いて評価すべき
根拠、または本当のギャップの所在)。再現できた場合は、次段の差別化ポイントをより確度高く
立てられる。

### 4. ギャップの特定と差別化

`research/research-gap.md` に書く:

- ギャップの言明: 「既存研究は A を達成しているが、B の条件下では C ができていない」の形式
- そのギャップが**解く価値がある**理由(objective.md の目的と接続する)
- そのギャップが**まだ解かれていない**理由(難しさの本質)
- 自分の研究の差別化ポイント候補
- `baseline-validation.md` で検証を行った場合はその結果を反映する(検証していない主張を
  そのまま前提にしない)

### 5. 仮説の立案と優先順位付け

**まず発散させる。** 特定したコアギャップ(絞り込んだ 1〜2 個程度)について、原因・メカニズムの異なる仮説を最低 3 つ立ててから比較する。「モデル容量が足りない」「データ分布が偏っている」「評価指標が実態を捉えていない」のように、説明の性質が異なる候補を並べること。同じ仮説の言い換えを 3 つ並べない(research-04 の手法発散と同じ考え方。最初に思いついた 1 つに飛びつくと、検証してから的外れだったと気づくコストが大きい)。

`research/hypotheses.md` に、各仮説を検証可能な形式で書く:

```markdown
## 仮説 H1: (1行の言明)
- 根拠: なぜ成り立つと考えるか(文献・理論・予備知見)
- 検証方法: どの実験・指標で真偽が判定できるか
- 棄却条件: どんな結果が出たら仮説を捨てるか
- 期待インパクト / 実現可能性(constraints.md と照合): 高・中・低
```

インパクト × 実現可能性で優先順位を付け、最優先の 1〜2 個を「本研究で検証する仮説」として推奨する。
**却下した仮説も理由とともに残す**(却下理由は research-04 の不採用案説明や research-12 の想定 Q&A の材料になる)。
最終決定はユーザーに AskUserQuestion で確認する。

### 6. 完了処理

- `research/STATUS.md` を更新(採択した仮説を記録。中核ベースラインの再現検証を行った場合はその結論も1行で記録)
- 次のステップ `/research-04-method-design` を案内

## 品質基準

- 「まだ誰もやっていない」だけでは価値にならない。「やる価値があるのに、難しくてできていない」ことを示す
- 仮説は複数(最低 3 つ、性質の異なるもの)を出してから絞り込む。最初に思いついた 1 つに飛びつかない
- 仮説は反証可能な形式で書く。「〜を改善する」ではなく「条件 X で指標 Y が Z 以上変化する」
- 文献に基づく主張には必ず出典(related-works.md のエントリ)を付ける
- ギャップの前提になっている中核的な主張は、鵜呑みにせず可能な範囲で再現検証する(検証しない場合も、しない判断をしたことを記録に残す)

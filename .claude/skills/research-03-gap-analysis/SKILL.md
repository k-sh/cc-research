# ギャップ分析と仮説立案

文献調査の結果から「まだ解かれていない、解く価値のある課題」を特定し、検証可能な研究仮説に落とし込む。

## 前提

開始時に `research/STATUS.md`・`research/literature-review.md`・`research/related-works.md`・`research/objective.md`・`research/constraints.md` を読む。
文献調査が無い場合は、ユーザーに既知の先行研究状況を質問するか、先に `/research-02-literature-review` の実行を提案する。

## 進め方

### 1. 課題の抽出

2 つの経路で未解決課題を集める:

- **明示的な限界**: 各論文の Limitation / Future Work に書かれていること
- **暗黙のギャップ**: 論文には書かれていないが比較表から読み取れること
  - どの手法も評価していない条件・データ・スケール
  - 手法カテゴリ間の未探索の組み合わせ
  - 研究設定と実運用条件の乖離(レイテンシ、データ品質、運用コスト)

### 2. 課題の構造化

抽出した課題を分類して `research/technical-issues.md` に整理する:

- **技術課題**: アルゴリズム・モデル・計算量の限界
- **実運用課題**: デプロイ・スケール・保守・コストの問題
- **評価課題**: 既存ベンチマーク・指標の不備

複数の論文に共通する課題は「分野の共通課題」として強調する。

### 3. ギャップの特定と差別化

`research/research-gap.md` に書く:

- ギャップの言明: 「既存研究は A を達成しているが、B の条件下では C ができていない」の形式
- そのギャップが**解く価値がある**理由(objective.md の目的と接続する)
- そのギャップが**まだ解かれていない**理由(難しさの本質)
- 自分の研究の差別化ポイント候補

### 4. 仮説の立案と優先順位付け

`research/hypotheses.md` に、各仮説を検証可能な形式で書く:

```markdown
## 仮説 H1: (1行の言明)
- 根拠: なぜ成り立つと考えるか(文献・理論・予備知見)
- 検証方法: どの実験・指標で真偽が判定できるか
- 棄却条件: どんな結果が出たら仮説を捨てるか
- 期待インパクト / 実現可能性(constraints.md と照合): 高・中・低
```

インパクト × 実現可能性で優先順位を付け、最優先の 1〜2 個を「本研究で検証する仮説」として推奨する。
最終決定はユーザーに AskUserQuestion で確認する。

### 5. 完了処理

- `research/STATUS.md` を更新(採択した仮説を記録)
- 次のステップ `/research-04-method-design` を案内

## 品質基準

- 「まだ誰もやっていない」だけでは価値にならない。「やる価値があるのに、難しくてできていない」ことを示す
- 仮説は反証可能な形式で書く。「〜を改善する」ではなく「条件 X で指標 Y が Z 以上変化する」
- 文献に基づく主張には必ず出典(related-works.md のエントリ)を付ける

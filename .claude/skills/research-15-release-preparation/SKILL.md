# リリース準備(Release Preparation)

研究成果(コード・モデル・データ)を公開できる状態に整える。

## 前提

開始時に `research/STATUS.md`・`paper/paper.md`・`research/reproducibility.md`・`research/experiment-guide.md` を読み、リポジトリの現状(既存 README・LICENSE の有無)を確認する。
ユーザーに確認する: 公開範囲(コードのみ/モデル込み/データ込み)、所属組織の公開ポリシーの有無。

## 進め方

### 1. 公開前の安全確認(最初にやる)

- シークレットのスキャン: APIキー・トークン・認証情報が履歴含めて含まれていないか(`git log -p` の確認や `gitleaks` 等)
- 個人情報・社外秘データの混入確認
- データ・依存ライブラリ・流用コードのライセンスが公開と両立するか

**研究中に参照・流用した外部コードのライセンス再点検。** `research/oss-references.md`
(research-07-implementation-plan でアーキテクチャの参考にした OSS)、`research/baseline-validation.md`
(research-03-gap-analysis で再現検証のために動かした公式実装)など、研究中に触れた外部コードを
洗い出す。**研究・内部検証目的での参照は許容されていても、公開は「広く再配布・改変・商用利用
されうる」用途への切り替えであり、同じコードでもライセンス上の意味合いが変わる。**

- コピーレフト系(GPL 等)のコードを一部でも組み込んでいないか。組み込んでいれば、これから
  選ぶ配布ライセンスと両立するか確認する(両立しなければ、そのコード片は自作で書き直す)
- 「研究・非商用利用限定」「引用の上でのみ利用可」など、再配布を制限する条項が付いていないか
- ライセンス表記・著作権表示が不明瞭なコード(断片的な転記、出典不明のコード片等)を含んでいないか
- 該当するコードが見つかった場合、**そのまま公開しない**。アルゴリズム・設計の理解だけを引き継ぎ、
  コード自体は自分の言葉で書き直す(参照元を見ながらの写経ではなく、理解してから何も見ずに書く)。
  書き直した箇所とその理由は `release-checklist.md` に記録する

- 問題が見つかったら公開作業を止めてユーザーに報告する

### 2. ライセンス選定

依存関係のライセンスを調べたうえで、候補(MIT / Apache-2.0 / BSD-3 / GPL 系 / CC 系)の違いを簡潔に説明し、AskUserQuestion でユーザーに選んでもらう。選定後 `LICENSE` を配置する。

### 3. ドキュメント整備

```
release/(またはリポジトリ直下。ユーザーに確認)
    README.md            # 概要、論文リンク、インストール、最小実行例、主要結果の表、再現手順への参照、引用方法、ライセンス
    MODEL_CARD.md        # モデル概要、学習データ、性能(evaluation-report.md より)、想定用途と範囲外用途、限界・バイアス、環境負荷(分かれば)
    DATA_CARD.md         # データ概要、収集方法、前処理、分割、ライセンス、既知の偏り(データ公開時のみ)
    CITATION.cff         # 引用情報(著者・タイトル・年・URL/DOI)
    RELEASE_NOTES.md     # バージョン、含まれるもの、既知の問題
    release-checklist.md # 公開チェックリスト
```

- README の実行例は**実際に動くコマンド**を書く(可能なら実行して確認する)
- 数値・主張は paper.md / evaluation-report.md からの転記とし、新しい主張を作らない
- サンプル(最小のデモスクリプトやノートブック)が無ければ作成を提案する

### 4. 公開チェックリスト

`release-checklist.md` に ✅/⬜ 付きで整理:

- [ ] シークレット・個人情報のスキャン済み
- [ ] LICENSE 配置済み・依存ライセンスと矛盾なし
- [ ] 研究中に参照した外部コード(oss-references.md・baseline-validation.md 等)のライセンスを再点検し、非両立なものは書き直し済み
- [ ] README の手順をクリーン環境で確認
- [ ] CITATION.cff が論文情報と一致
- [ ] 不要ファイル(実験ゴミ・巨大ファイル)の除去、.gitignore 整備
- [ ] モデル/データの配布方法(Git LFS・外部ホスティング)決定
- [ ] リポジトリ説明文・トピックタグ設定

### 5. 完了処理

- `research/STATUS.md` の 14 にチェックを入れ、パイプライン完了を宣言
- 残チェック項目(ユーザー自身の確認が必要なもの)を明示して報告する

## 品質基準

- 公開は不可逆(公開後に消しても取得され得る)。安全確認を省略しない
- 研究段階で許容されていたライセンス(研究・非商用利用限定、コピーレフト等)が、公開後も同じ
  意味を持つとは限らない。「研究中に問題なく動いていた」ことは公開してよい根拠にならない
- 「研究コードなので動かなくてもよい」としない。README の Quickstart は必ず通る状態にする

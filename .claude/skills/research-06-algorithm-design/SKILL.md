# アルゴリズム設計

採用手法を、実装者が迷わず書ける水準のアルゴリズムとして定式化する。

## 前提

開始時に `research/STATUS.md`・`research/selected-method.md`・`research/evaluation-metrics.md` を読む。
無い場合は手法の概要をユーザーに質問する。

## 進め方

### 1. 仕様の一貫性チェック(あいまいさ・矛盾の洗い出し)

数式・擬似コードに落とし込む前に、`selected-method.md`・`evaluation-metrics.md` を読み、
定式化に必要な情報が本当に揃っているかを確認する:

- **あいまいな点**: 中核アイデアの記述だけでは一意に定式化できない箇所(例: ある操作の
  エッジケースでの挙動、複数の定義がありうる正規化・集約方法)
- **矛盾・競合**: `selected-method.md` の記述と `evaluation-metrics.md` の指標定義が噛み合わない、
  想定インターフェースが `constraints.md` の資源制約と両立しない、など

見つかった場合、自分の判断で決め打ちして定式化を進めない。**具体的な選択肢(トレードオフ込み)を
提示し、`AskUserQuestion` でユーザーにどの方針で進めるか確認する。** ここでのあいまいさは、
後工程(`research-07-implementation` の詳細設計)にそのまま伝播し、実装段階で初めて発覚すると
手戻りが大きい。

### 2. 数学的定式化

- **記法表(Notation)を最初に作る**: 全記号の定義を一箇所に集約。以降すべての文書・論文でこの記法を使う
- 問題設定: 入力・出力・目的関数・制約を数式で定義
- 提案手法の中核を数式で表現(LaTeX 記法、`$$ ... $$`)
- 各式に「直観的な意味」を 1 行添える

### 3. アルゴリズム記述

- 擬似コード: Algorithm ブロック形式(入力/出力/手順を番号付きで)。学習アルゴリズムと推論アルゴリズムを分けて書く
- 各ステップに対応する数式番号を参照させる
- 停止条件・エッジケース(空入力、数値不安定になる条件)を明記

### 4. フロー・データフロー

- 全体フローチャートとデータフロー図を Mermaid で作成(`flowchart TD` など)
- 学習時と推論時でパスが違う場合は両方描く

### 5. 計算量解析

- 時間計算量・空間計算量を主要変数(データ数 n、次元 d 等)で導出する。**結果だけでなく導出過程を書く**
- ベースライン手法の計算量と対比する表を作る
- constraints.md の計算資源で実行可能かを見積もる(概算のメモリ量・実行時間)

### 6. 出力

```
research/
    algorithm-design.md   # 記法表、問題設定、定式化、計算量解析、設計上の判断とその理由
    pseudo-code.md        # 学習・推論の擬似コード(実装単位に対応)
    flowchart.md          # Mermaid によるフローチャート・データフロー図
```

### 7. 完了処理

- `research/STATUS.md` を更新
- 次のステップ `/research-07-implementation` を案内

## 品質基準

- 「実装者(自分を含む)がこの文書だけで実装に着手できるか」を完成基準とする
- selected-method.md の構成要素(アブレーション単位)と擬似コードのモジュール分割を一致させる
- 数式とコードで記号・変数名の対応表を作っておく(実装フェーズと論文執筆で効く)
- 近似・仮定を置いた箇所は、その妥当性と破れる条件を明記する
- 仕様書間のあいまいさ・矛盾は自分で解釈を決め打ちせず、ユーザーに方針を確認してから定式化に反映する

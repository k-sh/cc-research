# 文献調査(Literature Review)

既存研究を体系的に調査し、関連研究の全体像・ベースライン候補・比較表を作成する。

## 前提

開始時に `research/STATUS.md`・`research/research-plan.md`(調査キーワード)・`research/objective.md` を読む。
無い場合はエラーにせず、調査トピックと目的をユーザーに 1 回だけ質問して進める。

## 進め方

### 1. 検索

WebSearch / WebFetch を使う(未ロードなら ToolSearch で取得)。調査キーワードごとに以下の情報源を回る:

- **arXiv / Google Scholar**: 学術論文。`site:arxiv.org` などの検索演算子を活用
- **GitHub**: 実装・ベンチマークリポジトリ(スター数と最終更新日を記録)
- **企業テックブログ・公式ドキュメント**: 実運用知見
- **サーベイ論文**: 見つかったら優先的に読み、その参照リストから展開する

検索のコツ:
- 直近 1〜2 年の最新研究と、被引用数の多い基礎研究の両方を押さえる
- 日本語キーワードと英語キーワードの両方で検索する
- キーワードが多く並列に調査できる場合は、`model: haiku` を指定した general-purpose サブエージェントに検索を分担させてコストと時間を節約してよい

**検索ログを記録する。** 後で「この分野は本当に調べ尽くしたか」を検証可能にし、research-03 のギャップ分析やレビュー対応で「探索が甘い」と指摘されないための予防措置。使った検索式・情報源・日付・ヒット件数・そこから採用した文献数を `related-works.md` の末尾か `literature-review.md` に一覧化する。0 件だった検索式や、タイトルだけ見て除外した文献とその理由も残す(「探したが見つからなかったこと」は品質基準にもある通りギャップ分析の材料になる)。

**サブエージェントに調査結果のファイル書き込みをさせないこと。** 複数のサブエージェントが同時に走ると、
それぞれが `research/related-works.md` 等の同一ファイルに独立して書き込もうとし、互いの変更を
上書きして内容が消失・破損する(競合)。安全な分担方法は次のとおり:

1. 各サブエージェントには「特定のキーワード/情報源を調査し、見つかった文献を回答テキストとして返す」
   という調査タスクのみを与える。ファイルへの書き込みは指示しない
2. 全サブエージェントの回答が出揃うまで待ち、本体(このスキルを実行している自分自身)が結果を
   検証・統合したうえで、次の「3. 整理・出力」でファイル書き込みを単独・一括で行う

### 2. 引用の実在検証(重要)

**記憶から論文を書き出さない。** 一覧に載せる論文は、検索結果または実ページで存在を確認できたものだけにする。
各文献について以下を記録する。確認できなかった項目は空欄ではなく「未確認」と書く:

- タイトル / 著者 / 年 / 発表先(会議・ジャーナル・arXiv)
- URL または DOI(必須。実際にアクセスまたは検索結果で存在を確認したものだけを記録する)
- 被引用数の目安と確認日
- 一言要約・手法の特徴・長所・短所・自分の研究との関係

同じ文献を著者表記やプレプリント/正式版の違いで重複登録しないよう、タイトルまたは DOI で突き合わせて統合する。

### 3. 整理・出力

```
research/
    literature-review.md   # 調査の全体像: 調査方法、分類体系、時系列の流れ、主要な発見
    related-works.md       # 文献ごとの詳細カード + 分類別の比較表(手法/データ/指標/結果/長所/短所)
    baseline-summary.md    # ベースライン候補: 再現可能性(コード有無)、計算コスト、比較の適切さ
```

- 関連研究は 3〜6 個のカテゴリに分類し、カテゴリごとに共通点・相違点を 1 段落で総括する
- 比較表は「後で論文の Related Work にそのまま流用できる」粒度で作る
- ベースラインは「公式実装があるか」「自分の環境で動かせるか」を明記し、推奨順位を付ける
  (`research-03-gap-analysis` で、ギャップの前提になる中核ベースラインの再現検証に使う)

### 4. 完了処理

- `research/STATUS.md` の 02 にチェックを入れ、主要な発見(有力ベースライン、目立った未解決課題の気配)を追記
- 要約をユーザーに提示し、次のステップ `/research-03-gap-analysis` を案内

## 品質基準

- 文献数の目安は主要トピックあたり 10〜20 本。ただし数より「分類の納得感」と「ベースラインの実用性」を優先
- 一次情報(論文本体・公式リポジトリ)を確認せずに二次情報(ブログの紹介記事)だけで評価しない
- 調査で分かった「探したが見つからなかったこと」も記録する(ギャップ分析の材料になる)

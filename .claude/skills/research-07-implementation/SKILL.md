# 実装設計(Implementation Plan)

アルゴリズム設計を、実装可能な設計書と実装計画に落とし込む。

## 前提

開始時に `research/STATUS.md`・`research/algorithm-design.md`・`research/pseudo-code.md`・`research/constraints.md`・`research/evaluation-metrics.md` を読む。
既存コードベースがある場合はその構成・規約を先に把握し、設計を合わせる。

## 進め方

### 1. 全体アーキテクチャ

- ディレクトリ構成(データ、モデル、学習、推論、評価、設定、テスト、スクリプトの分離)
- モジュール分割: **pseudo-code.md のアルゴリズム単位と 1:1 に対応させる**(アブレーションで個別に差し替え可能にする)
- 設定管理方針: ハイパーパラメータ・パスは設定ファイル(YAML 等)に集約し、コードにハードコードしない
- 実験管理方針: 実行ごとに設定・シード・結果・ログを保存する仕組み(出力ディレクトリ規約)

### 2. 詳細設計

- 主要クラス・関数のシグネチャ(入出力の型・形状。テンソルは shape を明記)
- データ構造・データセットのスキーマ
- 学習ループ・推論パイプラインの設計(チェックポイント、早期終了、ロギング)
- ベースライン実行との共通インターフェース(同じ評価コードで提案手法とベースラインを測れる形)

### 3. テスト計画

- **ユニットテスト**: 数式どおりに動くかの検証(小さな手計算可能な入力で期待値一致)
- **形状・数値健全性テスト**: NaN/Inf 検出、shape 検証
- **統合テスト**: 極小データでの end-to-end 実行(学習 1〜2 step が通るか)
- **再現性テスト**: 同一シードで同一結果になるか
- **過学習サニティチェック**: 極小データに完全フィットできるか(できなければバグ)

### 4. CI・実装順序

- CI 設計: lint、ユニットテスト、極小データでの統合テストを自動実行
- 実装優先順位: MVP(仮説の signal が取れる最小構成)→ ベースライン再現 → 提案手法フル → アブレーション対応、のマイルストーンで刻む。各マイルストーンに完了条件を付ける

### 5. タスク分解(Task Breakdown)

各マイルストーンを、実装者が1回の作業で終えられる粒度の具体的なタスクに分解する。この分解が
粗いままだと(「MVPを実装する」のような大きさのタスクのまま)、次節のサブエージェントへの
実装委譲がうまく機能しない。委譲先は過去の会話を持たないため、タスクの範囲があいまいだと
何をすべきか自己流に解釈してしまう。

各タスクに書くこと:
- **ID**: `T1`, `T2` のような通し番号(依存関係の記述に使う)
- **内容**: pseudo-code.md のどのアルゴリズム単位、architecture.md のどのモジュールに
  対応するか。1〜2文で「何を作るか」を言い切る
- **Boundary(境界)**: このタスクが変更してよいファイル・モジュールの範囲。範囲外への変更は
  レビューで指摘対象にする(`references/code-review.md` の「担当外のモジュールへの越境」観点)
- **Depends(依存)**: 先に完了している必要のあるタスクID(無ければ「なし」)
- **完了条件**: test-plan.md のどのテストが通ればこのタスクは完了とみなすか

`research/tasks.md` に出力する。末尾に空の `## Implementation Notes` セクションを用意して
おく(実装を進める過程で得られた横断的な学びをここに追記していく。詳細は次節)。

### 6. 実装の進め方(タスクごとのサブエージェント委譲)

タスクが用意できたら、1タスクずつ次の3役をサブエージェントとして起動しながら進める。
過去の会話コンテキストは渡さず、タスク単体の情報だけで完結させる(長時間の対話を続けると
文脈が積み重なって判断が甘くなりやすいので、タスクごとに新しい目で取り組ませる):

- **実装者**: そのタスクの Task Brief(tasks.md の該当タスク + 参照すべき設計書の該当箇所 +
  `## Implementation Notes`)だけを渡して起動する。TDD で実装させる: 先に test-plan.md の
  対応するテストを書いて red(失敗)を確認させ、そのあと実装して green(成功)にする
  (`references/code-review.md` の RED-phase 証拠と同じ考え方)
- **レビューア**: `references/code-review.md` の Stage 1 をそのまま使う(独立サブエージェント、
  設計適合・バグ・境界チェック)
- **デバッガー**: レビューアが同じ指摘で2回連続 reject した場合に発動する
  (`references/code-review.md` の共通ルールを参照。最大2ラウンド)

1タスクが完了したら:
- 横断的な学び(ハマりどころ、他タスクにも関係する発見)があれば `tasks.md` の
  `## Implementation Notes` に追記する。次のタスクの実装者はこれを読んでから着手するので、
  同じ落とし穴を繰り返さずに済む
- 専用ブランチにコミットする(タスク単位が最小のコミット粒度になる。`main` や作業中の
  ブランチに直接コミットしない)
- 1マイルストーン分のタスクが終わったら、ダミーデータでの軽い動作確認結果を、図があれば
  それを埋め込んだ HTML(Artifact tool が使える環境ならそれで公開)でユーザーに提示し、
  合意を得てから次のマイルストーンに進む

1イテレーション1タスクで進めると、中断しても `tasks.md` の完了状態から「どこまで終わったか」
が分かり、再開しやすい。

対話的にユーザーと相談しながら進めたい場合は、この委譲を省略し自分がその場で書いても構わない
(タスクが小さい、または設計判断がまだ流動的な段階ではこちらの方が速いこともある)。ただし
その場合も、レビューア(Stage 1〜3)は必ず独立させる。

### 7. 統合検証(GO/NO-GO)

全タスク・全マイルストーンの実装が完了したら、`/research-08-evaluation` に進む前に、実装
全体が仕様通りに組み合わさっているかを一度通しで検証する。個別タスクのレビューを通過して
いても、タスク間のインターフェースの食い違い(境界は守っていても、別のタスクが期待する
形式でデータを渡していない、等)は見落とされがちなため。

独立したサブエージェントに、`tasks.md`(完了状態・Implementation Notes)・`architecture.md`・
`test-plan.md`・実際のコードを渡し、次を確認させる:
- 個別タスクではなく全テストスイートをまとめて実行した結果、通るか
- 設計書上つながっているはずのモジュール間のインターフェースが実際に合っているか
- 仕様(hypotheses.md・evaluation-metrics.md)への対応漏れがないか

判定は次の3値で返させる:
- `GO`: 問題なし。次のステップに進んでよい
- `NO-GO`: 修正が必要な問題がある。該当タスクの実装に戻ってやり直す
- `MANUAL_VERIFY_REQUIRED`: 自動では判断しきれない(実環境依存、手動確認が必要な項目など)。
  該当項目を明示してユーザーに確認を依頼する

### 8. 出力

```
research/
    implementation-plan.md  # マイルストーン、優先順位、実験管理方針、リスク
    architecture.md         # ディレクトリ構成、モジュール設計、クラス/API 設計、データ構造
    test-plan.md            # テスト一覧(目的・入力・期待値)、CI 設計
    tasks.md                # タスク一覧(ID・内容・Boundary・Depends・完了条件)、
                             # Implementation Notes
```

### 9. 完了処理

- 実装に着手した場合は、統合検証(GO/NO-GO)が `GO` であることを確認してから
  `research/STATUS.md` を更新する
- 実装自体に着手するか確認し、次の検証ステップとして `/research-08-evaluation` を案内
- 実際の実験コードを書く際は、書いたそばから自己判断だけで正しさを確認しない。
  `research-08-evaluation/references/code-review.md` にある3段階の独立レビュー(過去の
  会話コンテキストを渡さないサブエージェントによる、設計適合・バグのレビュー → ダミー
  データでの妥当性検証 → 実データ適用後の結果妥当性レビュー)を、test-plan.md のテスト
  計画と合わせて適用する

## 品質基準

- 「ベースラインを先に動かす」を原則とする(提案手法の評価はベースラインの再現ができてから)
- 実験の設定・結果が後から追跡できること(reproducibility フェーズを楽にする)を設計段階で織り込む
- 研究コードでも、評価に関わる部分(指標計算・データ分割)はテストを必須にする
- 複数ステップから成る実装は、段階を踏んで進める。各段階でダミーデータによる軽い動作確認を
  行い、うまくいっていることをユーザーと確認してから次の段階に進む(一気に最後まで実装して
  から初めて動かす、という進め方をしない)
- 実装は専用ブランチで進め、段階ごとにコミットする。品質に問題ないことをユーザーと合意して
  から元のブランチにマージする(未検証のコードをいきなり主要ブランチに混ぜない)
- タスクは Boundary(担当範囲)を明確にし、担当外のモジュールに手を入れない。範囲があいまいな
  タスクは、独立した実装者・レビューアに委譲したときに機能しない
- 全タスク完了後は統合検証(GO/NO-GO)を経てから次のステップへ進む。個別タスクが全て
  レビューを通過していても、タスク間のインターフェースが噛み合っているとは限らない

# 実装設計(Implementation Plan)

アルゴリズム設計を、実装可能な設計書と実装計画に落とし込む。

## 前提

開始時に `research/STATUS.md`・`research/algorithm-design.md`・`research/pseudo-code.md`・`research/constraints.md`・`research/evaluation-metrics.md` を読む。
既存コードベースがある場合はその構成・規約を先に把握し、設計を合わせる。

## 進め方

### 1. 全体アーキテクチャ

- ディレクトリ構成(データ、モデル、学習、推論、評価、設定、テスト、スクリプトの分離)
- モジュール分割: **pseudo-code.md のアルゴリズム単位と 1:1 に対応させる**(アブレーションで個別に差し替え可能にする)
- 設定管理方針: ハイパーパラメータ・パスは設定ファイル(YAML 等)に集約し、コードにハードコードしない
- 実験管理方針: 実行ごとに設定・シード・結果・ログを保存する仕組み(出力ディレクトリ規約)

### 2. 詳細設計

- 主要クラス・関数のシグネチャ(入出力の型・形状。テンソルは shape を明記)
- データ構造・データセットのスキーマ
- 学習ループ・推論パイプラインの設計(チェックポイント、早期終了、ロギング)
- ベースライン実行との共通インターフェース(同じ評価コードで提案手法とベースラインを測れる形)

### 3. テスト計画

- **ユニットテスト**: 数式どおりに動くかの検証(小さな手計算可能な入力で期待値一致)
- **形状・数値健全性テスト**: NaN/Inf 検出、shape 検証
- **統合テスト**: 極小データでの end-to-end 実行(学習 1〜2 step が通るか)
- **再現性テスト**: 同一シードで同一結果になるか
- **過学習サニティチェック**: 極小データに完全フィットできるか(できなければバグ)

### 4. CI・実装順序

- CI 設計: lint、ユニットテスト、極小データでの統合テストを自動実行
- 実装優先順位: MVP(仮説の signal が取れる最小構成)→ ベースライン再現 → 提案手法フル → アブレーション対応、のマイルストーンで刻む。各マイルストーンに完了条件を付ける
- **マイルストーンを一括で実装してから通しで確認しない。** 全部書き上げてから初めて動かすと、
  どこで壊れたかの切り分けが難しくなり、手戻りが大きくなる。1マイルストーン(できれば
  さらに小さい、意味のある単位)ごとに実装 → ダミーデータ(極小データ・手計算できる入力)で
  軽く動作確認 → 結果をユーザーに見せて次に進んでよいか合意を得る、というサイクルで進める。
  この単位はそのまま `references/code-review.md` の Stage 1(実装中レビュー)の粒度にもなる
- **確認を依頼する際、図(学習曲線・出力サンプルなど)がある場合はテキストの羅列だけで
  済ませない。** 図を埋め込んだ HTML を作成して提示する(Artifact tool が使える環境なら
  それで公開し、無ければファイルに保存してパスを案内する)。数値やパスの羅列だけでは
  状態が伝わりにくく、確認の質が落ちる
- **実装は `main`(または現在の作業ブランチ)に直接コミットせず、専用のブランチを切って行う。**
  各段階(マイルストーン、または `references/code-review.md` の Stage 1 を1回通した単位)が
  終わるたびにコミットする。一気に実装してから一つの巨大なコミットにまとめない。全マイルストーン
  が終わり、品質(独立レビュー・ダミーデータ検証)に問題がないことをユーザーと合意できてから、
  元のブランチにマージする。マージ前のブランチは、後から「どこで壊れたか」を `git bisect` 等で
  追う際の頼りにもなる

### 5. 出力

```
research/
    implementation-plan.md  # マイルストーン、優先順位、実験管理方針、リスク
    architecture.md         # ディレクトリ構成、モジュール設計、クラス/API 設計、データ構造
    test-plan.md            # テスト一覧(目的・入力・期待値)、CI 設計
```

### 6. 完了処理

- `research/STATUS.md` を更新
- 実装自体に着手するか確認し、次の検証ステップとして `/research-08-evaluation` を案内
- この設計書をもとに実際の実験コードを書き始める場合は、書いたそばから自己判断だけで
  正しさを確認しない。`research-08-evaluation/references/code-review.md` にある3段階の
  独立レビュー(過去の会話コンテキストを渡さないサブエージェントによる、設計適合・バグの
  レビュー → ダミーデータでの妥当性検証 → 実データ適用後の結果妥当性レビュー)を、
  test-plan.md のテスト計画と合わせて適用する

## 品質基準

- 「ベースラインを先に動かす」を原則とする(提案手法の評価はベースラインの再現ができてから)
- 実験の設定・結果が後から追跡できること(reproducibility フェーズを楽にする)を設計段階で織り込む
- 研究コードでも、評価に関わる部分(指標計算・データ分割)はテストを必須にする
- 複数ステップから成る実装は、段階を踏んで進める。各段階でダミーデータによる軽い動作確認を
  行い、うまくいっていることをユーザーと確認してから次の段階に進む(一気に最後まで実装して
  から初めて動かす、という進め方をしない)
- 実装は専用ブランチで進め、段階ごとにコミットする。品質に問題ないことをユーザーと合意して
  から元のブランチにマージする(未検証のコードをいきなり主要ブランチに混ぜない)

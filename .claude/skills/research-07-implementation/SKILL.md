# 詳細設計(Detailed Design)

アルゴリズム設計を、実装者(人間または独立したサブエージェント)がそのまま着手できる水準の
詳細設計書・タスク一覧に落とし込む。**このスキルはコードを書かない。** 何をどう作るかを
具体的に決め切ることが役割で、実際に手を動かして実装するのは次の `research-08-implementation-execution`
の仕事。役割を分けることで、設計判断とコーディング作業がひとつのスキルに肥大化するのを防ぐ。

## 前提

開始時に `research/STATUS.md`・`research/algorithm-design.md`・`research/pseudo-code.md`・`research/constraints.md`・`research/evaluation-metrics.md` を読む。
既存コードベースがある場合はその構成・規約を先に把握し、設計を合わせる。

## 進め方

### 1. 全体アーキテクチャ

- ディレクトリ構成(データ、モデル、学習、推論、評価、設定、テスト、スクリプトの分離)
- モジュール分割: **pseudo-code.md のアルゴリズム単位と 1:1 に対応させる**(アブレーションで個別に差し替え可能にする)
- 設定管理方針: ハイパーパラメータ・パスは設定ファイル(YAML 等)に集約し、コードにハードコードしない
- 実験管理方針: 実行ごとに設定・シード・結果・ログを保存する仕組み(出力ディレクトリ規約)

### 2. 詳細設計

- 主要クラス・関数のシグネチャ(入出力の型・形状。テンソルは shape を明記)
- データ構造・データセットのスキーマ
- 学習ループ・推論パイプラインの設計(チェックポイント、早期終了、ロギング)
- ベースライン実行との共通インターフェース(同じ評価コードで提案手法とベースラインを測れる形)

### 3. テスト計画

- **ユニットテスト**: 数式どおりに動くかの検証(小さな手計算可能な入力で期待値一致)
- **形状・数値健全性テスト**: NaN/Inf 検出、shape 検証
- **統合テスト**: 極小データでの end-to-end 実行(学習 1〜2 step が通るか)
- **再現性テスト**: 同一シードで同一結果になるか
- **過学習サニティチェック**: 極小データに完全フィットできるか(できなければバグ)

### 4. CI・実装順序

- CI 設計: lint、ユニットテスト、極小データでの統合テストを自動実行
- 実装優先順位: MVP(仮説の signal が取れる最小構成)→ ベースライン再現 → 提案手法フル → アブレーション対応、のマイルストーンで刻む。各マイルストーンに完了条件を付ける

### 5. タスク分解(Task Breakdown)

各マイルストーンを、実装者が1回の作業で終えられる粒度の具体的なタスクに分解する。この分解が
粗いままだと(「MVPを実装する」のような大きさのタスクのまま)、`research-08-implementation-execution`
でのサブエージェントへの実装委譲がうまく機能しない。委譲先は過去の会話を持たないため、タスクの
範囲があいまいだと何をすべきか自己流に解釈してしまう。

各タスクに書くこと:
- **ID**: `T1`, `T2` のような通し番号(依存関係の記述に使う)
- **内容**: pseudo-code.md のどのアルゴリズム単位、architecture.md のどのモジュールに
  対応するか。1〜2文で「何を作るか」を言い切る
- **Boundary(境界)**: このタスクが変更してよいファイル・モジュールの範囲。範囲外への変更は
  実装実行時のレビューで指摘対象にする
- **Depends(依存)**: 先に完了している必要のあるタスクID(無ければ「なし」)
- **完了条件**: test-plan.md のどのテストが通ればこのタスクは完了とみなすか

`research/tasks.md` に出力する。末尾に空の `## Implementation Notes` セクションを用意して
おく(`research-08-implementation-execution` が実装を進める過程で得られた横断的な学びを
ここに追記していく)。

### 6. 出力

```
research/
    implementation-plan.md  # マイルストーン、優先順位、実験管理方針、リスク
    architecture.md         # ディレクトリ構成、モジュール設計、クラス/API 設計、データ構造
    test-plan.md            # テスト一覧(目的・入力・期待値)、CI 設計
    tasks.md                # タスク一覧(ID・内容・Boundary・Depends・完了条件)、
                             # Implementation Notes(空)
```

### 7. 完了処理

- `research/STATUS.md` を更新
- 次のステップとして `/research-08-implementation-execution`(タスクの実装実行)を案内

## 品質基準

- 「ベースラインを先に動かす」を原則とする(提案手法の評価はベースラインの再現ができてから)
- 実験の設定・結果が後から追跡できること(reproducibility フェーズを楽にする)を設計段階で織り込む
- 研究コードでも、評価に関わる部分(指標計算・データ分割)はテストを必須にする
- タスクは Boundary(担当範囲)を明確にし、依存関係(Depends)を過不足なく書く。範囲や依存が
  あいまいなタスクは、独立した実装者・レビューアに委譲したときに機能しない
- 「実装者(自分を含む、または独立したサブエージェント)がこの設計書とタスク一覧だけで
  実装に着手できるか」を完成基準とする

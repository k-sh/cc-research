# 実験コードの独立レビュー(3段階)

研究実験のコードは「動いた」ことと「意図通りに動いた」ことが一致しない場合が多い
(指標の定義ミス、データリーク、シード未固定、統計検定の選択ミスなどは実行時エラーにならない)。
これらは実装した本人(=このスキルを実行している自分)が読み返しても見落としやすい。
そのため、実験に使うコードを書いた・変更した際は、**過去の会話コンテキストを渡さない
独立したサブエージェント**にレビューさせ、指摘があれば直してから先に進む。

## 共通ルール

- レビューは Agent tool でサブエージェント(`general-purpose` 想定)を起動して行う。
- サブエージェントには、それまでの会話・設計判断の経緯を渡さない。渡すのは
  「参照すべき仕様書のパス」「対象コードのパス」「レビューさせたい観点」のみ。
- プロンプトに「これで合っているはず」「一応確認して」のような期待値・誘導を書かない。
  独立した第三者として虚心にジャッジさせる。
- 指摘(特に結論を覆しうる指摘)が出たら、コードを修正した上で**同じ段階のレビューをやり直す**。
  指摘を読んだだけで次の段階に進まない。
- 些末な指摘(命名・スタイル等)は直すかどうかを自分で判断してよい。致命的な指摘
  (指標計算・データ分割・判定ロジックの誤りなど、結論に影響し得るもの)は必ず解消する。

## Stage 1: 実装中レビュー(仕様適合・バグ)

**いつ**: 学習ループ・指標計算・データ分割・統計検定など、結果を左右するコードを書いた直後、
実行する前。

**渡す情報**: 対象コードのパス、`research/evaluation-metrics.md`・`research/hypotheses.md`・
`research/baseline-summary.md`・`research/implementation-plan.md`(あれば `architecture.md`)。

**観点**:
- 指標の定義・計算方法が仕様書と一致しているか(平均の取り方、正負の向き、正規化の有無など)
- データ分割・前処理にリークがないか(train/test 汚染、正規化統計の計算範囲など)
- 乱数シードの固定、比較条件(ベースラインと提案手法で同じ前処理・同じ探索予算か)
- shape 不整合、NaN/Inf の発生源、例外の握りつぶし
- 「動くが仕様と違うことをしている」パターン(引数の順序違い、指標の別定義を実装してしまった等)

**プロンプト例**(そのまま流用可、パスは実際のものに置換):

```
あなたはこのコードの作成には関与していない第三者のレビュアーです。
以下の設計書とコードを読み、コードが設計書の意図通りに実装されているか、
また典型的なバグ(データリーク、shape不整合、指標計算ミス、シード未固定、比較条件の不揃い等)
がないかを客観的にレビューしてください。

設計書: {evaluation-metrics.md, hypotheses.md, baseline-summary.md のパス}
コード: {対象ファイルのパス}

指摘がある場合は、ファイル・箇所・何が期待と違うかを具体的に挙げてください。
問題が見当たらない場合はその旨明記してください。「動くはず」ではなく「仕様と一致しているか」
の観点で判断してください。
```

## Stage 2: 完成時レビュー(ダミーデータでの妥当性検証)

**いつ**: コードが一通り完成し、実データにかける前。

**やること**(実行主体である自分が行う):

1. 仕様(`test-plan.md`・`evaluation-metrics.md`)から、性質の異なる2種類のダミーテストセットを作る。
   - **成功するはずのケース**: 提案手法が明確に検出できるはずのシグナルを持つ小規模データ
     (例: ベースラインより明らかに良い/悪いことが分かっている合成データ)。
   - **絶対に失敗する・差が出ないはずのケース**: ラベルシャッフル、完全ランダムなデータ、
     提案手法とベースラインを同一入力にした比較など、「有意差が出てはいけない」ことが
     分かっているケース。
2. 両ケースで実際にコードを実行し、出力・ログを残す。
3. 独立したサブエージェントに、コード・両ケースの入力と出力を渡し判定させる。

**観点**:
- 「成功するはず」のケースで、本当に期待通りの結果(シグナルの検出、有意差の検出)になっているか。
  たまたま通っているだけ(閾値が緩すぎる等)ではないか。
- 「失敗するはず」のケースで、コードが正しく「差がない/失敗」と判定できているか。
  逆に、バグにより見かけ上「成功」と誤判定していないか(これが最も見落とされやすい)。
- 有意差判定・閾値・pass/fail の基準そのものが妥当か(仕様の判定基準と一致しているか)。

**プロンプト例**:

```
あなたはこのコードの作成には関与していない第三者のレビュアーです。
以下は研究実験コードに対して作成した2種類のテストケースと、その実行結果です。

コード: {対象ファイルのパス}
ケースA(成功するはずのシグナルを持つダミーデータとその結果): {パス・出力}
ケースB(有意差が出ない/失敗するはずのダミーデータとその結果): {パス・出力}
判定基準の仕様: {evaluation-metrics.md 等のパス}

以下を判定してください:
1. ケースAは本当に「成功」と判定されるべき結果になっているか
2. ケースBは正しく「差がない/失敗」と判定されているか。バグにより誤って「成功」と
   判定されていないか
3. 判定ロジック(閾値・検定の選び方など)自体が仕様と整合しているか
問題があれば具体的に指摘してください。
```

指摘があれば、コードを修正して **Stage 1 からやり直す**(判定ロジックの修正はコード変更を伴うため)。

## Stage 3: 実データ適用後レビュー(結果の妥当性)

**いつ**: 実データで実験を実行し、結果が出た後、`evaluation-report.md` の結論を確定する前。

**渡す情報**: `research/hypotheses.md`、事前登録した実験計画、`research/experiment-results.md`
(生データ)、`evaluation-report.md` のドラフト(結果サマリと考察)。

**観点**:
- 「うまくいった」場合: その改善が本物の効果によるものか、それともコードのバグ・データリーク・
  比較条件の不公平さ・シード数不足など別の要因で説明できてしまわないか。
- 「うまくいかなかった」場合: 手法自体に効果がないのか、それとも実装のバグ・設定ミス・
  探索予算の不均衡が原因で正しく評価できていないだけなのか。
- シード数・統計検定・効果量の報告が、その結論を支えるのに十分な強さか。
- 好都合な結果だけを見て、都合の悪い実行結果(失敗ログなど)を無視していないか。

**プロンプト例**:

```
あなたはこの実験の実施には関与していない第三者のレビュアーです。
以下の仮説・実験計画・生の実験結果・導かれた結論を読み、結論の妥当性を検証してください。

仮説と実験計画: {hypotheses.md, evaluation-report.md 冒頭の事前登録部分}
生データ: {experiment-results.md}
導かれた結論: {evaluation-report.md の考察部分}

「うまくいった」結論であれば、それがコードのバグ・データリーク・比較条件の不公平さなど
別の要因で説明できてしまわないか検討してください。
「うまくいかなかった」結論であれば、それが手法自体の限界なのか、実装上の問題で
正しく評価できていないだけなのかを切り分けてください。
結論を支える証拠(シード数・検定・効果量)が十分かも評価してください。
```

指摘が出た場合:
- コードの問題が疑われる → 該当箇所を修正し **Stage 1 からやり直す**(実験の再実行が必要になる)。
- 手法自体の限界と判断される → `evaluation-report.md` の考察・限界セクションに正直に反映する
  (悪い結果を隠さないという `research-08-evaluation` の品質基準そのもの)。

## 記録

各段階のレビューで致命的な指摘が出た場合は、何を指摘され・どう対応したかを
`research/code-review-log.md` に1エントリだけ短く残す(仕様書ではないので詳細な議事録は不要、
後から「なぜこの実装になったか」を追える程度でよい)。指摘なしで通過した場合は記録不要。

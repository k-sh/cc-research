# 実験コードの独立レビュー(3段階)

研究実験のコードは「動いた」ことと「意図通りに動いた」ことが一致しない場合が多い
(指標の定義ミス、データリーク、シード未固定、統計検定の選択ミスなどは実行時エラーにならない)。
これらは実装した本人(=このスキルを実行している自分)が読み返しても見落としやすい。
そのため、実験に使うコードを書いた・変更した際は、**過去の会話コンテキストを渡さない
独立したサブエージェント**にレビューさせ、指摘があれば直してから先に進む。

## 共通ルール

### 対象とする変更の粒度

レビュー対象は「結果を左右するコード」に限る: 学習ループ、指標計算、データ分割、統計検定、
判定ロジック(pass/fail・有意差判定)など。ログ文言の変更、コメント修正、変数名のリネームなど
結果に影響しない変更は対象外でよい。全ての瑣末な変更にフルの3段階を毎回回すと、実験のイテレーション
速度を損なうコストに見合わなくなる。目安として、Stage 1 は上記コアロジックを新規作成・変更するたびに、
Stage 2/3 は「実データに投入する前」「結果が出てレポートを確定する前」という節目で1回でよい。

### レビューの実行方法

- レビューは Agent tool でサブエージェント(`general-purpose` 想定)を起動して行う。
- サブエージェントには、それまでの会話・設計判断の経緯を渡さない。渡すのは
  「参照すべき仕様書のパス」「対象コードのパス」「レビューさせたい観点」のみ。
- **モデル**: 実装者(このスキル自身)と同格のモデルでレビューすると、同種の見落としを
  再生産しやすい。Agent tool の `model` パラメータで明示的に指定する。Stage 1・2 は `sonnet`
  以上、判断の難度が最も高い Stage 3(結果解釈の妥当性)は既定で `opus` を使う。
- プロンプトに「これで合っているはず」「一応確認して」のような期待値・誘導を書かない。
  独立した第三者として虚心にジャッジさせる。

### 出力フォーマット(構造化)

サブエージェントには自由記述ではなく、次の形式で厳密に返させる。曖昧な文章判定は
親エージェント側での見落とし・誤読を招くため、機械的に次の一手を判断できる形にする
(下記の各プロンプト例には、この指定をそのまま含めてある):

```
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容(該当ファイル・箇所・何が期待と違うか)
(指摘がない場合は「指摘: なし」と明記)
```

### 指摘への対応

- 重大度「高」(結論を覆しうるもの: 指標計算・データ分割・判定ロジックの誤りなど)は
  必ず解消してから次に進む。「中」「低」は対応するか、対応せず記録だけして先に進むかを
  自分で判断してよい。
- 指摘を解消したら**同じ段階のレビューをやり直す**。指摘を読んだだけで次の段階に進まない。
- **レビュアーの判定は絶対ではない。** 指摘の根拠が薄い、または明確に反証できる場合は、
  盲目的に修正せず、指摘とその反証をユーザーに提示して判断を仰ぐ。
- **同一箇所について3回(目安。状況に応じて調整してよい)修正・再レビューしても重大度「高」の
  指摘が解消しない場合、または複数回のレビューで判定が割れる場合は、レビューを打ち切って
  ユーザーに状況(指摘内容・試した修正・現状)を報告し判断を仰ぐ。** 無限にレビュー→修正を
  繰り返さない。

## Stage 1: 実装中レビュー(仕様適合・バグ)

**いつ**: 学習ループ・指標計算・データ分割・統計検定など、結果を左右するコードを書いた直後、
実行する前。

**渡す情報**: 対象コードのパス、`research/evaluation-metrics.md`・`research/hypotheses.md`・
`research/baseline-summary.md`・`research/implementation-plan.md`(あれば `architecture.md`)。

**観点**:
- 指標の定義・計算方法が仕様書と一致しているか(平均の取り方、正負の向き、正規化の有無など)
- データ分割・前処理にリークがないか(train/test 汚染、正規化統計の計算範囲など)
- 乱数シードの固定、比較条件(ベースラインと提案手法で同じ前処理・同じ探索予算か)
- shape 不整合、NaN/Inf の発生源、例外の握りつぶし
- 「動くが仕様と違うことをしている」パターン(引数の順序違い、指標の別定義を実装してしまった等)
- **設計との対応関係を項目ごとに確認する。** 「全体として大体合っていそう」という印象だけの
  判断は見落としを生みやすい。設計書(evaluation-metrics.md の各指標、implementation-plan.md/
  architecture.md の主要コンポーネント)を1項目ずつ拾い、対応するコード上の場所(ファイル・
  関数/クラス名)が実在するかを確認させる。**設計書にあるのに実装されていない項目**、逆に
  **設計書にない独自の処理**(仕様にない補正・丸め・条件分岐など)がないかを洗い出す

**プロンプト例**(そのまま流用可、パスは実際のものに置換):

```
あなたはこのコードの作成には関与していない第三者のレビュアーです。
以下の設計書とコードを読み、コードが設計書の意図通りに実装されているか、
また典型的なバグ(データリーク、shape不整合、指標計算ミス、シード未固定、比較条件の不揃い等)
がないかを客観的にレビューしてください。

設計書: {evaluation-metrics.md, hypotheses.md, baseline-summary.md のパス}
コード: {対象ファイルのパス}

「動くはず」ではなく「仕様と一致しているか」の観点で判断してください。
設計書に書かれている項目(指標の定義、主要なコンポーネント・処理ステップ)を1つずつ拾い、
コード上の対応箇所が実在するかを確認してください。設計書にあるのに実装されていない項目、
逆に設計書にない独自の処理を追加してしまっている箇所があれば指摘してください。

出力は次の形式で厳密に返してください:
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容(該当ファイル・箇所・何が期待と違うか)
(指摘がない場合は「指摘: なし」と明記)
```

## Stage 2: 完成時レビュー(ダミーデータでの妥当性検証)

**いつ**: コードが一通り完成し、実データにかける前。

**やること**(実行主体である自分が行う):

1. 仕様(`test-plan.md`・`evaluation-metrics.md`)から、性質の異なる2種類のダミーテストセットを作る。
   - **成功するはずのケース**: 提案手法が明確に検出できるはずのシグナルを持つ小規模データ
     (例: ベースラインより明らかに良い/悪いことが分かっている合成データ)。
   - **絶対に失敗する・差が出ないはずのケース**: ラベルシャッフル、完全ランダムなデータ、
     提案手法とベースラインを同一入力にした比較など、「有意差が出てはいけない」ことが
     分かっているケース。
2. 両ケースで実際にコードを実行し、出力・ログを残す。
3. 独立したサブエージェントに、コード・両ケースの入力と出力を渡し判定させる。

**観点**:
- 「成功するはず」のケースで、本当に期待通りの結果(シグナルの検出、有意差の検出)になっているか。
  たまたま通っているだけ(閾値が緩すぎる等)ではないか。
- 「失敗するはず」のケースで、コードが正しく「差がない/失敗」と判定できているか。
  逆に、バグにより見かけ上「成功」と誤判定していないか(これが最も見落とされやすい)。
- 有意差判定・閾値・pass/fail の基準そのものが妥当か(仕様の判定基準と一致しているか)。
- **テストケース自体の設計が妥当か。** 閾値が緩すぎてバグがあっても通ってしまわないか、
  シグナルが弱すぎて成功ケースが本当の検証になっていないか、成功ケースが簡単すぎて
  そもそもバグを検出できない作りになっていないかも合わせて判定させる(テストケースを
  作った本人にはこの甘さが見えにくい)。

**プロンプト例**:

```
あなたはこのコードの作成には関与していない第三者のレビュアーです。
以下は研究実験コードに対して作成した2種類のテストケースと、その実行結果です。

コード: {対象ファイルのパス}
ケースA(成功するはずのシグナルを持つダミーデータとその結果): {パス・出力}
ケースB(有意差が出ない/失敗するはずのダミーデータとその結果): {パス・出力}
判定基準の仕様: {evaluation-metrics.md 等のパス}

以下を判定してください:
1. ケースAは本当に「成功」と判定されるべき結果になっているか
2. ケースBは正しく「差がない/失敗」と判定されているか。バグにより誤って「成功」と
   判定されていないか
3. 判定ロジック(閾値・検定の選び方など)自体が仕様と整合しているか
4. ケースA・ケースBのテストケース自体の設計は妥当か(閾値が緩すぎてバグを見逃さないか、
   シグナルが弱すぎて/強すぎて検証になっていないか)

出力は次の形式で厳密に返してください:
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容(該当ファイル・箇所・何が期待と違うか)
(指摘がない場合は「指摘: なし」と明記)
```

指摘があれば、コードを修正して **Stage 1 からやり直す**(判定ロジックの修正はコード変更を伴うため)。

## Stage 3: 実データ適用後レビュー(結果の妥当性)

**いつ**: 実データで実験を実行し、結果が出た後、`evaluation-report.md` の結論を確定する前。

**渡す情報**: `research/hypotheses.md`、事前登録した実験計画、`research/experiment-results.md`
(生データ)、`evaluation-report.md` のドラフト(結果サマリと考察)。

**観点**:
- 「うまくいった」場合: その改善が本物の効果によるものか、それともコードのバグ・データリーク・
  比較条件の不公平さ・シード数不足など別の要因で説明できてしまわないか。
- 「うまくいかなかった」場合: **まず実装バグを疑う。** 手法が新しいほど「効果がないのが真実」
  よりも「どこかにバグがある」方が圧倒的に多い。手法自体の限界と結論する前に、**最も単純な
  ケース**で改めて実行し、そこで期待通りの妥当な結果が出るかを再評価する。ここでいう
  「単純」とは、差が最も大きく出るように設計したケース(Stage 2 の「成功するはずのケース」は
  信号検出を狙って作られており、必ずしも単純とは限らない)ではなく、**手計算・目視で正解が
  分かるくらい小さく作った入力**(test-plan.md のユニットテスト・過学習サニティチェックと
  同じ規模感)を指す。最も単純なケースですら期待通りにならない場合はほぼ確実に実装側の問題
  であり、複雑な実データでの「効果なし」をそのまま手法の結論として採用しない
- シード数・統計検定・効果量の報告が、その結論を支えるのに十分な強さか。
- **多重比較の補正。** 複数指標・複数条件・複数シードにわたって有意差検定を行っている場合、
  Bonferroni 補正等を考慮しているか、それとも単一の有意水準を使い回して第一種の過誤
  (偶然の有意差を実際の効果と誤認するリスク)を膨らませていないか。
- 好都合な結果だけを見て、都合の悪い実行結果(失敗ログなど)を無視していないか。

**プロンプト例**:

```
あなたはこの実験の実施には関与していない第三者のレビュアーです。
以下の仮説・実験計画・生の実験結果・導かれた結論を読み、結論の妥当性を検証してください。

仮説と実験計画: {hypotheses.md, evaluation-report.md 冒頭の事前登録部分}
生データ: {experiment-results.md}
導かれた結論: {evaluation-report.md の考察部分}

「うまくいった」結論であれば、それがコードのバグ・データリーク・比較条件の不公平さなど
別の要因で説明できてしまわないか検討してください。
「うまくいかなかった」結論であれば、まず実装バグの可能性を疑ってください。最も単純な
ケース(差が最も大きく出るケースではなく、手計算・目視で正解が分かるくらい小さい入力)で
改めて妥当な結果が出るか確認したかを問い、確認していなければその点を指摘してください。
それでも手法自体の限界と言えるのかを切り分けてください。
結論を支える証拠(シード数・検定・効果量)が十分かも評価してください。
複数の指標・条件・シードにわたって検定を行っている場合、多重比較の補正が必要かどうか、
必要なら考慮されているかも確認してください。

出力は次の形式で厳密に返してください:
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容(該当ファイル・箇所・何が期待と違うか)
(指摘がない場合は「指摘: なし」と明記)
```

### 統計だけでなく、出力そのものを見る

統計指標(平均・有意差・効果量)が良くても、それだけでは「本当に目的を達成しているか」を
判断しきれないことが多い。指標上は改善していても、生成物を実際に見るとモード崩壊・不自然な
繰り返し・意味不明な出力・テンプレートの丸暗記といった兆候が見えることがある。逆に指標が
ぱっとしなくても、出力自体は目的を捉えていることもある(指標の選び方が的外れな兆候)。
統計処理を経由しない、生の出力そのものに対する評価をここに追加する。

- 実行主体(自分)は、条件ごとに**代表的な出力サンプル**(良さそうな例・悪そうな例・指標が
  中央値付近の例を含む複数点)を用意し、`research/figures/samples/` 等、独立レビュアーが
  直接見て判断できる形式(画像、生成テキスト、予測と正解の対比など)で保存する
- 独立したサブエージェントに、**まず統計結果を見せずに**、目的(`research/objective.md`・
  `research/hypotheses.md`)とサンプル出力のパスだけを渡し、「これは目的を達成しているように
  見えるか」を判定させる。画像等は説明文で済まさず、Read ツールで直接開いて見られるように
  パスをそのまま渡す
- そのうえで統計結果(experiment-results.md・導かれた結論)も見せ、**見た目の判断と統計上の
  結論が一致するか**を確認させる。一致しない場合(統計は良いのに出力は悪く見える、その逆)は、
  指標の定義・集計方法・サンプルの選び方に問題がないかを疑う
- 出力が画像・テキストのような直接見られる artifact を持たないタスク(スカラー値のみ等)の
  場合は、この評価は該当なしとしてよい。ただしその場合も、予測と正解の対比表など何らかの
  「生の結果を直接見る」手段が本当に無いかを一度検討してから省略する

**プロンプト例(出力そのものの評価)**:

```
あなたはこの実験の実施には関与していない第三者のレビュアーです。
まだ統計結果は見せません。以下の目的と、実際に生成された出力サンプルを見て、
この出力が目的を達成しているように見えるかを判断してください。

目的: {objective.md, hypotheses.md のパス}
出力サンプル: {良さそう/悪そう/中央値付近の出力サンプルのパス一覧}

判断してください:
1. これらの出力は、目的に照らして「うまくいっている」ように見えるか
2. 見た目の質が疑わしい兆候(モード崩壊、不自然な繰り返し、意味不明な出力、テンプレートの
   丸暗記、明らかな失敗パターンなど)はないか

出力は次の形式で厳密に返してください:
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容
(指摘がない場合は「指摘: なし」と明記)
```

上記の判定を得たら、続けて統計結果(experiment-results.md・導かれた結論)を渡し、
「見た目の判断と統計上の結論は一致するか」を追加で判定させる。一致しない場合は重大度「高」の
指摘として扱う。

指摘が出た場合:
- コードの問題が疑われる → 該当箇所を修正し **Stage 1 からやり直す**(実験の再実行が必要になる)。
- 出力そのものに疑わしい兆候があるのに統計だけ見て「成功」と判断していた → 原因(指標の選択・
  サンプルの偏り・データの問題)を調査し、統計だけで結論を確定しない
- 「うまくいかなかった」結論で、最も単純なケースでの再評価をまだ行っていない → 先に
  手計算・目視で正解が分かる規模の最も単純なケース(Stage 2 の「成功するはずのケース」を
  そのまま使うのではなく、必要ならさらに単純化する)で妥当な結果が出るか確認してから、
  手法自体の限界かどうかを判断する
- 単純なケースでも妥当な結果が出ないことを確認した上で、なお手法自体の限界と判断される →
  `evaluation-report.md` の考察・限界セクションに正直に反映する(悪い結果を隠さないという
  `research-08-evaluation` の品質基準そのもの)。

## 記録

各段階のレビュー結果は、判定(PASS/FAIL)・指摘の要約・対応を1エントリだけ短く
`research/code-review-log.md` に残す(仕様書ではないので詳細な議事録は不要)。
指摘なしで一発 PASS した場合も「Stage n: PASS(指摘なし)」の1行は残す。
再現性フェーズ(`research-10-reproducibility`)で「独立レビューを実施済みか」を
確認する際の材料になる。

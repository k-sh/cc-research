# 実装タスクの独立レビュー(2段階)

自分(または委譲した実装者)が書いたコードは、動けば正しいと思い込みやすい。設計との
細かなズレ、担当外モジュールへの越境、「テストを書いたが実は何も検証していない」といった
問題は実行時エラーにならず静かに紛れ込む。そのため、タスクのコードを書いた・変更した際は、
**過去の会話コンテキストを渡さない独立したサブエージェント**にレビューさせ、指摘があれば
直してから先に進む。

**目次**: [共通ルール](#共通ルール)(粒度・実行方法・出力フォーマット・指摘への対応・
デバッガー・完了宣言前のフレッシュな証拠)/ [Stage 1](#stage-1-設計適合境界チェック)
(設計適合・境界チェック)/ [Stage 2](#stage-2-完成時レビューredgreen-の妥当性検証)
(RED→GREEN の妥当性検証)/ [記録](#記録)

## 共通ルール

### 対象とする変更の粒度

`research/tasks.md` の1タスク=1回の Stage 1 レビューが基本単位。タスクの中でも結果を左右する
コア部分(ロジック本体)が対象で、ログ文言の変更やコメント修正など結果に影響しない変更は
対象外でよい。

### レビューの実行方法

- レビューは Agent tool でサブエージェント(`general-purpose` 想定)を起動して行う。
- サブエージェントには、それまでの会話・設計判断の経緯を渡さない。渡すのは「参照すべき
  設計書のパス(`tasks.md` の該当タスク・`architecture.md`・`test-plan.md`)」「対象コードの
  パス」「レビューさせたい観点」のみ。
- **モデル**: 実装者(このスキル自身、または委譲した実装者サブエージェント)と同格のモデルで
  レビューすると、同種の見落としを再生産しやすい。Agent tool の `model` パラメータで
  明示的に指定する(`sonnet` 以上)。
- プロンプトに「これで合っているはず」「一応確認して」のような期待値・誘導を書かない。
  独立した第三者として虚心にジャッジさせる。

### 出力フォーマット(構造化)

サブエージェントには自由記述ではなく、次の形式で厳密に返させる:

```
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容(該当ファイル・箇所・何が期待と違うか)
(指摘がない場合は「指摘: なし」と明記)
```

### 指摘への対応

- 重大度「高」(結論を覆しうるもの: ロジックの誤り、境界違反、インターフェース不整合など)は
  必ず解消してから次に進む。「中」「低」は対応するか、対応せず記録だけして先に進むかを
  自分で判断してよい。
- 指摘を解消したら**同じ段階のレビューをやり直す**。指摘を読んだだけで次の段階に進まない。
- **レビュアーの判定は絶対ではない。** 指摘の根拠が薄い、または明確に反証できる場合は、
  盲目的に修正せず、指摘とその反証をユーザーに提示して判断を仰ぐ。
- **同じ指摘で2回連続 reject されたら、闇雲に3回目の修正を試みない。** 代わりに
  「デバッガー」役の独立したサブエージェントを起動する(過去の会話コンテキストを渡さない。
  原因調査に必要なら WebSearch/WebFetch の使用も許可する)。デバッガーには、指摘内容・
  これまで試した修正・関連コードを渡し、次を返させる:
  - `ROOT_CAUSE`: 何が根本原因か
  - `CATEGORY`: 実装バグ/仕様理解のズレ/テスト自体の誤り/タスク分解自体の問題(Boundary・
    Depends の設定ミス)、など
  - `FIX_PLAN`: 次に何をどう直すべきかの具体的な手順
  この fix plan に沿って修正し、レビューをやり直す。**デバッガーのラウンドは最大2回まで。**
  2回のデバッグでも同じ指摘が解消しない場合はレビューを打ち切り、ユーザーに状況(指摘内容・
  試した修正・デバッガーの調査結果)を報告して判断を仰ぐ。`CATEGORY` が「タスク分解自体の
  問題」に近い場合は、`research-07-implementation` に戻ってタスク分解をやり直すことを提案する。
  無限にレビュー→修正を繰り返さない。

### 完了宣言の前に(フレッシュな証拠)

「直った」「動いた」「PASS した」とは、記憶や以前の実行結果だけで言わない。必ずその場で
再実行し、新しく得られた証拠(出力・ログ・テスト結果)を確認してから宣言する。修正を加えた
直後に「これで直ったはず」で終わらせず、実際に再実行して確認できるまでは未検証(NOT_VERIFIED)
として扱い、ユーザーにも「未検証」と「検証済み」を混同させない。

## Stage 1: 設計適合・境界チェック

**いつ**: タスクのコードを(実装者サブエージェントか自分が)書いた直後、次のタスクに進む前。

**渡す情報**: 対象コードのパス、`tasks.md` の該当タスク(内容・Boundary・Depends・完了条件)、
`architecture.md`・`test-plan.md` の関連箇所。

**観点**:
- タスクの「内容」通りに実装されているか。項目を1つずつ拾い、対応するコード上の場所が
  実在するかを確認する。**タスクの内容にあるのに実装されていない部分**、逆に**タスクに
  ないのに追加してしまった処理**がないか
- **Boundary を越えていないか。** このタスクが変更してよいファイル・モジュールの範囲を
  超えて、他のタスクが担当するモジュールにまで手を入れていないか
- **Depends 先のタスクとのインターフェースが噛み合っているか。** 依存先のタスクが渡す
  データの形式・型・shape と、このタスクが期待する形式が一致しているか
- shape 不整合、NaN/Inf の発生源、例外の握りつぶし

**機械的に確認できることは読解より先にコマンドで確認する**:
- `git diff` で実際の変更差分を見る(最終状態のコードだけを読んで「大体合っていそう」と
  判断しない)
- `TODO` / `FIXME` / `pass` だけのスタブ実装が残っていないか grep する
- 該当するテストを実際に実行し、通ることを確認する(通ったという報告を鵜呑みにしない)

**プロンプト例**(そのまま流用可、パスは実際のものに置換):

```
あなたはこのコードの作成には関与していない第三者のレビュアーです。
以下のタスク定義・設計書とコードを読み、コードがタスクの意図通りに実装されているか、
Boundary(担当範囲)を越えていないか、Depends先とのインターフェースが噛み合っているか、
また典型的なバグ(shape不整合、NaN/Inf、例外の握りつぶし等)がないかを客観的にレビューして
ください。

タスク定義: {tasks.md の該当タスク}
設計書: {architecture.md, test-plan.md の関連箇所}
コード: {対象ファイルのパス}

タスクの内容を1つずつ拾い、コード上の対応箇所が実在するかを確認してください。タスクに
あるのに実装されていない部分、逆にタスクにない独自の処理を追加してしまっている箇所、
Boundary を越えた変更があれば指摘してください。

出力は次の形式で厳密に返してください:
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容(該当ファイル・箇所・何が期待と違うか)
(指摘がない場合は「指摘: なし」と明記)
```

## Stage 2: 完成時レビュー(RED→GREEN の妥当性検証)

**いつ**: タスクの実装が完成し、次のタスクに進む前。

**やること**(実行主体である自分、または委譲した実装者が行う):

1. **RED-phase の証拠を残す。** test-plan.md の対応するテストを、実装前(またはスタブ化した
   状態)で実行し、「失敗する」ことを一度確認してログに残す。これを飛ばして最初からテストが
   パスしている場合、そのテストは実は何も検証していない可能性がある
2. 実装し、同じテストを実行して「成功する」(GREEN)ことを確認する。出力・ログを残す
3. 独立したサブエージェントに、コード・テスト・RED 時/GREEN 時のログを渡し判定させる

**観点**:
- RED-phase の証拠があるか。無い場合、テストが本当にバグを検出できる設計かどうか誰も
  確認していないことになる
- GREEN になった理由が実装の正しさによるものか、それともテスト自体を緩めてパスさせて
  いるだけではないか(閾値を緩める、アサーションを削る、など)
- テストケース自体が、手計算・目視で期待値と一致することを確認できる規模・入力になっているか

**プロンプト例**:

```
あなたはこのコードの作成には関与していない第三者のレビュアーです。
以下はタスクのテストと、その RED 時・GREEN 時の実行結果です。

コード: {対象ファイルのパス}
テスト: {対応するテストファイルのパス}
RED-phase の証拠(実装前・スタブ状態での実行結果): {ログ}
GREEN の証拠(実装後の実行結果): {ログ}

以下を判定してください:
1. RED-phase の証拠は、テストが本当に失敗すべき理由で失敗しているか(無関係なエラーで
   落ちているだけではないか)
2. GREEN になった理由は実装の正しさによるものか。テストの閾値・アサーションを緩めて
   通しているだけではないか
3. テストケース自体が、手計算・目視で期待値と一致することを確認できる規模か

出力は次の形式で厳密に返してください:
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容(該当ファイル・箇所・何が期待と違うか)
(指摘がない場合は「指摘: なし」と明記)
```

指摘があれば、コードを修正して **Stage 1 からやり直す**。

## 記録

各段階のレビュー結果は、判定(PASS/FAIL)・指摘の要約・対応を1エントリだけ短く
`research/code-review-log.md` に残す(仕様書ではないので詳細な議事録は不要)。指摘なしで
一発 PASS した場合も「Stage n: PASS(指摘なし)」の1行は残す。デバッガーが介入した場合は
`ROOT_CAUSE`/`CATEGORY`/`FIX_PLAN` の要約も残す。

他のタスクにも関わる横断的な学びが見つかった場合は、`code-review-log.md` だけに留めず、
`research/tasks.md` の `## Implementation Notes` にも転記する。後続タスクの実装者はここを
読んでから着手するため、同じ落とし穴を踏まずに済む。

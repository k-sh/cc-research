# 実装実行(Implementation Execution)

`research-07-implementation-plan` が作った `tasks.md` を、実際に動くコードにする。

## 前提

開始時に `research/STATUS.md`・`research/tasks.md`・`research/implementation-plan.md`・
`research/architecture.md`・`research/test-plan.md` を読む。`tasks.md` が無い場合は、
先に `/research-07-implementation-plan` の実行を提案する。`tasks.md` の `## Implementation Notes`
に既知のハマりどころが無いか確認する。

## 進め方

### 1. タスクごとのサブエージェント委譲

`tasks.md` の未完了タスクを Depends の順に1つずつ処理する。1タスクにつき次の3役を
サブエージェントとして起動する。過去の会話コンテキストは渡さず、タスク単体の情報だけで
完結させる(長時間の対話を続けると文脈が積み重なって判断が甘くなりやすいので、タスクごとに
新しい目で取り組ませる):

- **実装者**: そのタスクの Task Brief(`tasks.md` の該当タスク + 参照すべき設計書の該当箇所 +
  `## Implementation Notes`)だけを渡して起動する。TDD で実装させる: 先に test-plan.md の
  対応するテストを書いて red(失敗)を確認させ、そのあと実装して green(成功)にする
  (`references/code-review.md` の RED-phase 証拠と同じ考え方)
- **レビューア**: `references/code-review.md` の Stage 1(独立サブエージェント、設計適合・
  バグ・境界チェック)を使う
- **デバッガー**: レビューアが同じ指摘で2回連続 reject した場合に発動する
  (`references/code-review.md` の共通ルールを参照。最大2ラウンド)

対話的にユーザーと相談しながら進めたい場合は、この委譲を省略し自分がその場で書いても構わない
(タスクが小さい、または設計判断がまだ流動的な段階ではこちらの方が速いこともある)。ただし
その場合も、レビューア(Stage 1・2)は必ず独立させる。

### 2. タスク完了時の処理

1タスクが完了したら:
- 横断的な学び(ハマりどころ、他タスクにも関係する発見)があれば `tasks.md` の
  `## Implementation Notes` に追記する。次のタスクの実装者はこれを読んでから着手するので、
  同じ落とし穴を繰り返さずに済む
- タスクの状態(未着手/進行中/完了)を `tasks.md` に反映する
- 専用ブランチにコミットする(タスク単位が最小のコミット粒度になる。`main` や作業中の
  ブランチに直接コミットしない)

1イテレーション1タスクで進めると、中断しても `tasks.md` の完了状態から「どこまで終わったか」
が分かり、再開しやすい。

### 3. マイルストーンの確認

1マイルストーン分のタスクが終わったら、ダミーデータ(極小データ・手計算できる入力)での
軽い動作確認結果を、図があればそれを埋め込んだ HTML(Artifact tool が使える環境ならそれで
公開)でユーザーに提示し、合意を得てから次のマイルストーンに進む。数値やパスの羅列だけでは
状態が伝わりにくい。

### 4. 統合検証(GO/NO-GO)

全タスク・全マイルストーンの実装が完了したら、`/research-09-evaluation` に進む前に、実装
全体が仕様通りに組み合わさっているかを一度通しで検証する。個別タスクのレビューを通過して
いても、タスク間のインターフェースの食い違い(境界は守っていても、別のタスクが期待する
形式でデータを渡していない、等)は見落とされがちなため。

独立したサブエージェントに、`tasks.md`(完了状態・Implementation Notes)・`architecture.md`・
`test-plan.md`・実際のコードを渡し、次を確認させる:
- 個別タスクではなく全テストスイートをまとめて実行した結果、通るか
- 設計書上つながっているはずのモジュール間のインターフェースが実際に合っているか
- 仕様(hypotheses.md・evaluation-metrics.md)への対応漏れがないか

判定は次の3値で返させる:
- `GO`: 問題なし。次のステップに進んでよい
- `NO-GO`: 修正が必要な問題がある。該当タスクの実装に戻ってやり直す
- `MANUAL_VERIFY_REQUIRED`: 自動では判断しきれない(実環境依存、手動確認が必要な項目など)。
  該当項目を明示してユーザーに確認を依頼する

### 5. 出力

```
research/
    tasks.md              # 完了状態・Implementation Notes を更新
    code-review-log.md    # 各タスクのレビュー結果(PASS/FAIL・指摘・対応)
```

実装コード自体は `architecture.md` の設計に従ったソースツリーに出力する(`research/` 配下ではない)。

### 6. 完了処理

- 統合検証が `GO` であることを確認してから `research/STATUS.md` を更新する
- 次のステップとして `/research-09-evaluation` を案内

## 品質基準

- 実装コードは自己レビューだけで進めない(`references/code-review.md` の独立レビューを飛ばさない)
- 複数ステップから成る実装は、段階を踏んで進める。各段階でダミーデータによる軽い動作確認を
  行い、うまくいっていることをユーザーと確認してから次の段階に進む(一気に最後まで実装して
  から初めて動かす、という進め方をしない)
- 実装は専用ブランチで進め、タスクごとにコミットする。品質に問題ないことをユーザーと合意して
  から元のブランチにマージする(未検証のコードをいきなり主要ブランチに混ぜない)
- タスクの Boundary(担当範囲)を越えて他モジュールに手を入れない
- 全タスク完了後は統合検証(GO/NO-GO)を経てから次のステップへ進む。個別タスクが全て
  レビューを通過していても、タスク間のインターフェースが噛み合っているとは限らない
- 「直った」「動いた」は記憶や以前の実行結果ではなく、その場で再実行した新しい証拠で宣言する

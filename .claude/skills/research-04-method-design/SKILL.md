# 手法設計(Method Design)

研究仮説を検証するための新手法を複数考案し、体系的に比較して採用案を決定する。

## 前提

開始時に `research/STATUS.md`・`research/hypotheses.md`・`research/research-gap.md`・`research/constraints.md`・`research/related-works.md` を読む。
仮説が無い場合は、解きたい課題をユーザーに質問して進める。

## 進め方

### 1. 解決案の発散(視点を分離して独立に生成する)

**同じ会話の流れで複数案を続けて書かない。** 1 つ目の案を書いた直後に 2 つ目を考えると、
表面上は別物でも発想の芯は 1 つ目に無意識に引っ張られ(アンカリング)、結局似た案が並ぶ。
これを防ぐため、案ごとに独立した subagent(Agent tool。過去の会話コンテキストを渡さず、
問題設定・制約・関連研究の要約と割り当てた視点だけを渡す)に並列で発想させる。視点の一覧・
渡す情報・集約後の罠検出は `references/divergent-ideation.md` を参照。

- 最低 3 視点を使う。うち少なくとも 1 つは「異分野転用」「前提の撤廃」のような意外性の高い
  視点にする(手堅い視点だけを 3 つ選ぶと結局似た案が並ぶ)
- 各 subagent には他の視点やそこで出た案を見せない。見せた時点でアンカリングが起きる
- 各案について: 中核アイデア(1〜2 文)、動作原理の直観、なぜ仮説を解決できるか、を書かせる

### 2. 体系的な比較評価

**まず罠(trap)を検出する。** 隔離発散で出た案には、一見魅力的でも致命的な欠陥を隠している
ものが紛れうる。`references/divergent-ideation.md` の観点(既知の失敗パターンの踏襲、実在しない
リソースの前提、constraints.md への気づかない違反)を確認し、該当する案は削除せず
「trap: 理由」を付けて残す。

各案を以下の軸で評価し、スコア表にまとめる(各軸 1〜5 + 根拠 1 行):

| 軸 | 見るポイント |
|---|---|
| 新規性 | related-works.md のどの既存手法とも本質的に異なるか |
| 実現可能性 | 期間・スキル・データで実装しきれるか |
| 計算コスト | 学習・推論のコストが constraints.md の資源に収まるか |
| 制約適合性 | ライセンス・データ・運用条件に反しないか |
| 期待効果 | 仮説の指標をどの程度改善しそうか(根拠付き) |
| リスク | 失敗する場合の最も可能性が高いシナリオ |

**新規性の主張は文献と突き合わせる。** 「〜と似た既存研究は無いか」を意識的に自問し、怪しければ追加検索する。

### 3. 採用案の決定

- 推奨案とその理由を提示し、AskUserQuestion でユーザーに最終決定してもらう
- **不採用案も破棄しない**: 不採用理由を記録する(論文の設計判断の説明や、査読対応・アブレーション設計で必ず使う)

### 4. 出力

```
research/
    method-proposals.md   # 全手法案の詳細 + 比較スコア表 + 不採用理由
    selected-method.md    # 採用案の詳細: 中核アイデア、構成要素、新規性の言明、
                          # 想定される弱点と対策、段階的な実装計画(MVP→フル)、将来拡張案
```

`selected-method.md` の「構成要素」は、後のアブレーション実験で個別に外せる単位に分解して書く。

### 5. 完了処理

- `research/STATUS.md` を更新(採用手法名と新規性の言明を記録)
- 次のステップとして `/research-05-patent-search`(知財確認)を案内。特許が関係しない純学術研究なら `/research-06-algorithm-design` へのスキップも提案する

## 品質基準

- 案は独立した subagent に隔離して発散させる。自分一人で続けて3案書くと、最初の案にアンカリングして結局似た案になりやすい
- 「動きそう」ではなく「なぜ動くのか」のメカニズムを書く
- 新規性の言明は 1 文で言えること(言えないなら差別化が弱い)
- MVP(最小構成)を定義し、最初の 2 週間で仮説の signal が得られる計画にする

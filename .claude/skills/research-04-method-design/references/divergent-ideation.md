# 隔離発散(Isolated Divergent Ideation)

研究上の分岐点(手法案・仮説の候補など、複数の妥当な答えがありうる場面)で、
「最初に思いついた案に後続の案が無意識に引っ張られる」現象(アンカリング)を防ぐための手順。

## なぜ隔離するか

同じ会話の流れの中で案を1つ書き、続けて2つ目・3つ目を書くと、表面上の言葉は変えても
発想の芯は1つ目に似てしまう。人間が「3つ挙げて」と言われて似た案を並べがちなのと同じ現象が、
モデルにも起きる。これを防ぐ最も確実な方法は、案ごとに完全に独立した文脈で発想させ、
互いを一切見せないことである(会話を分けても、同じ応答の中で連続して書かせては意味がない)。

## 手順

1. 下記の視点(フレーム)から、問題の性質に合わせて最低 3 つ選ぶ。**うち少なくとも 1 つは
   「異分野転用」「前提の撤廃」など意外性の高い視点にする。** 手堅い視点だけを 3 つ選ぶと、
   結局似たような案が並ぶ
2. 視点ごとに独立した subagent(Agent tool)を起動する。渡す情報は「問題設定(仮説や解きたい
   課題)・制約(constraints.md)・関連研究の要約」と「割り当てた視点」だけにする。**他の視点や
   そこで出た案は絶対に見せない。** 見せると、その時点でアンカリングが起きる
3. 各 subagent には、その視点から中核アイデアを 1〜2 個、それぞれ「中核アイデア(1〜2文)・
   動作原理の直観・なぜ課題を解決できるか」の形式で返させる。評価はさせない(生成と評価を
   混ぜると、評価者としての慎重さが発想の幅を狭める)
4. 全 subagent の結果が揃ってから、本体(このスキルを実行している自分自身)が一箇所に集約し、
   次工程の比較評価に渡す

## 視点(フレーム)一覧

| 視点 | 発想の起点 |
|---|---|
| **手堅い拡張** | 既存手法(中核ベースライン)の直接延長。前提は疑わず改善だけに集中する |
| **異分野転用** | 全く違う分野(生物学・物理学・経済学・制御理論・オペレーションズリサーチ等)の解法をそのまま持ち込んだらどうなるか |
| **前提の撤廃** | この問題設定で「当然」とされている制約・仮定を1つ選び、それが外れたら何ができるか |
| **逆転** | 通常のアプローチを反転させる(事前に防ぐ→起きてから直す、集中処理→分散処理、能動→受動 等) |
| **組み合わせ** | 複数の既存アプローチの境界をずらして混ぜる。単純な足し算ではなく、境界の引き直し方を変える |
| **極小構成($0・1時間)** | 予算も時間も無いとしたら、核心だけを満たす最も安上がりな案は何か |
| **極大構成(無制限)** | 計算資源・データ・人員が無制限だったらどう解くか。実現不可能でも、本質的に何が効くかのヒントになる |
| **査読者/攻撃者** | この手法を落とす・破綻させるとしたら査読者・攻撃者はどこを突くか。その弱点の裏から着想する |
| **運用現場** | 実運用でこの手法が壊れるとしたらどこか。壊れにくい設計から逆算する |
| **専門知識のない視点** | その分野の前提知識に染まっていない立場から見た、素朴だが的を射た疑問は何か |

## 集約後の罠(trap)検出

隔離発散で出た案は、一見魅力的でも致命的な欠陥を隠していることがある。比較評価(スコアリング)の
前に、必ず次を確認する:

- related-works.md にある既知の失敗パターンをそのまま踏襲していないか
- 実在しない、またはライセンス上使えないデータ・リソースを前提にしていないか
- constraints.md の制約(期間・予算・計算資源・秘密保持等)に気づかず違反していないか

該当する案は削除せず「trap: 理由」を付けて比較表に残す(採否の判断材料、および後で
「なぜこの案を採らなかったか」を説明する記録になる)。

# 実験結果の独立レビュー

アブレーション・感度分析のコードは、評価実験のコードを流用・改造して作ることが多く、
「要素を外す/置き換える」の実装ミス(実は外し忘れている、比較対象が評価実験と条件が
ズレている等)が結果を静かに歪める。ソースコード自体の設計適合レビューは
`research-08-implementation-execution` が担うため、ここでは重複させない。その代わり、
**実データで結果が出た後**、過去の会話コンテキストを渡さない独立したサブエージェントに、
寄与率・数値の結論と生の出力の両面から結果の妥当性をレビューさせる。結果に問題がある
(コードのバグが疑われる)場合は、前工程(実装)に戻って直し、再実行してからこのレビューを
やり直す。

**目次**: [共通ルール](#共通ルール)(実行方法・出力フォーマット・指摘への対応・デバッガー・
完了宣言前のフレッシュな証拠)/ [結果の妥当性レビュー](#結果の妥当性レビュー)(数値の結論・
出力そのものの目視評価)/ [記録](#記録)

## 共通ルール

### いつ行うか

実データでアブレーション・感度分析を実行し、結果が出た後、`ablation-study.md`・
`sensitivity-analysis.md` の結論を確定する前に必ず1回行う。瑣末な文言修正のたびに毎回回す
必要はないが、結果の解釈・結論に関わる変更をした場合は都度やり直す。

### レビューの実行方法

- レビューは Agent tool でサブエージェント(`general-purpose` 想定)を起動して行う。
- サブエージェントには、それまでの会話・設計判断の経緯を渡さない。渡すのは
  「参照すべき仕様書のパス」「結果データ・ドラフトのパス」「レビューさせたい観点」のみ。
- **モデル**: 判断の難度が高い(寄与の結論の妥当性・生の出力の質を横断的に評価する)ため、
  Agent tool の `model` パラメータで既定 `opus` を明示的に指定する。
- プロンプトに「これで合っているはず」「一応確認して」のような期待値・誘導を書かない。
  独立した第三者として虚心にジャッジさせる。

### 出力フォーマット(構造化)

サブエージェントには自由記述ではなく、次の形式で厳密に返させる:

```
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容(該当ファイル・箇所・何が期待と違うか)
(指摘がない場合は「指摘: なし」と明記)
```

### 指摘への対応

- 重大度「高」(結論を覆しうるもの)は必ず解消してから次に進む。「中」「低」は対応するか、
  対応せず記録だけして先に進むかを自分で判断してよい。
- **レビュアーの判定は絶対ではない。** 指摘の根拠が薄い、または明確に反証できる場合は、
  盲目的に修正せず、指摘とその反証をユーザーに提示して判断を仰ぐ。
- **同じ指摘で2回連続 reject されたら、闇雲に3回目の修正を試みない。** 代わりに
  「デバッガー」役の独立したサブエージェントを起動する(過去の会話コンテキストを渡さない。
  原因調査に必要なら WebSearch/WebFetch の使用も許可する)。デバッガーには、指摘内容・
  これまで試した修正・関連コードを渡し、次を返させる:
  - `ROOT_CAUSE`: 何が根本原因か
  - `CATEGORY`: 「効いていない」場合の原因分析の順番(下記)に対応させ、実装バグ/テストデータの
    不整合(座標系・単位・時間軸など)/環境要因(他プロセス・熱スロットリングなど)/先行研究に
    対する認識齟齬/**手法自体の論理の誤り(構成要素の分解や寄与の前提の問題)**、のいずれかで返す
  - `FIX_PLAN`: 次に何をどう直すべきかの具体的な手順
  この fix plan に沿って修正し、レビューをやり直す。**デバッガーのラウンドは最大2回まで。**
  2回のデバッグでも同じ指摘が解消しない場合はレビューを打ち切り、ユーザーに状況(指摘内容・
  試した修正・デバッガーの調査結果)を報告して判断を仰ぐ。無限にレビュー→修正を繰り返さない。
  特に `CATEGORY` が「手法自体の論理の誤り」に近いと判断された場合は、実装を直し続けるのではなく
  `research-04-method-design`・`research-06-algorithm-design` への差し戻しをユーザーに提案する。

### 完了宣言の前に(フレッシュな証拠)

「直った」「うまくいった」「PASS した」とは、記憶や以前の実行結果だけで言わない。必ず
その場で再実行し、新しく得られた証拠(出力・ログ・テスト結果)を確認してから宣言する。
修正を加えた直後に「これで直ったはず」で終わらせず、実際に再実行して確認できるまでは
未検証(NOT_VERIFIED)として扱い、ユーザーにも「未検証」と「検証済み」を混同させない。

## 結果の妥当性レビュー

**いつ**: 実データでアブレーション・感度分析を実行し、結果が出た後、`ablation-study.md`・
`sensitivity-analysis.md` の結論を確定する前。

**渡す情報**: `research/selected-method.md`、条件表(事前登録した比較条件)、
`research/experiment-results.md`(生データ)、`ablation-study.md`/`sensitivity-analysis.md` のドラフト。

**観点**:
- 「要素 X が効いている」という結論の場合: それが本物の寄与によるものか、それとも
  条件の揃え忘れ・シード数不足・比較コードのバグで説明できてしまわないか。
- 「要素 X は効いていない」という結論の場合: **原因分析は次の順番で行い、手法そのものを疑うのは
  最後の手段とする。** 「寄与が本当に小さい」よりも他の原因である方が多い。次の順で1つずつ
  消していく:
  1. **実装を疑う。** 「外したつもりが外れていない」等の実装バグ、使用ライブラリ・APIの
     使い方の認識齟齬がないか。**最も単純なケース**(差が最大に出るように設計したケースでは
     なく、手計算・目視で正解が分かるくらい小さく作った入力)で改めて実行し、期待通りの差が
     出るか確認する
  2. **テストデータを疑う。** 特に複数のデータソースを組み合わせている場合、一部だけ時間軸・
     座標系・単位が異なっている、といったデータ由来の不整合がないか確認する
  3. **環境を疑う。** 特に処理速度・メモリ使用量など効率面の指標が想定と違う場合、同一サーバ上の
     他プロセスや熱スロットリングなど、コード自体とは無関係な環境要因がないか確認する
  4. **方法に対する認識齟齬を疑う。** `selected-method.md` でこの要素になぜ寄与があると
     想定したのか(参考にした先行研究があればそれも)を読み直し、想定と実装がずれていないか
     確認する
  5. **方法そのものを疑う。** ここまでの4つを消してもなお説明がつかない場合に初めて、
     この要素は実際に寄与が小さいと判断する

  この順番を飛ばして「寄与が小さい」と結論しない。最も単純なケースですら差が出ない場合は
  ほぼ確実に1〜4のいずれかが原因
- 寄与の大きさが、下がり方のメカニズム(仮説)と整合しているか、それとも数字だけ見て
  こじつけていないか。
- **多重比較の補正。** アブレーションは条件数が多くなりがちで、条件ごとに有意差検定を
  行うと第一種の過誤(偶然の差を実際の寄与と誤認するリスク)が積み上がる。Bonferroni 補正等が
  必要か、必要なら考慮されているかを確認する。
- 好都合な条件の結果だけを見て、都合の悪い結果(相互作用が疑われる組み合わせなど)を
  無視していないか。

**プロンプト例**:

```
あなたはこの実験の実施には関与していない第三者のレビュアーです。
以下の構成要素の分解・比較条件・生の実験結果・導かれた結論を読み、結論の妥当性を検証してください。

構成要素と比較条件: {selected-method.md, 条件表}
生データ: {experiment-results.md}
導かれた結論: {ablation-study.md / sensitivity-analysis.md の該当部分}

「要素Xが効いている/効いていない」という結論が、比較条件の不揃い・シード数不足・
実装のバグなど別の要因で説明できてしまわないか検討してください。「効いていない」結論であれば、
次の順で原因分析が行われているか確認してください: (1) 実装バグ・ライブラリ/APIの誤用、
(2) テストデータの不整合(座標系・単位・時間軸など)、(3) 環境要因(他プロセス・熱スロット
リングなど)、(4) 先行研究に対する認識齟齬、(5) 手法そのものの限界。この順番のうち確認が
飛ばされているステップがあれば指摘してください。最も単純なケース(差が最も大きく出るケース
ではなく、手計算・目視で正解が分かるくらい小さい入力)での再評価は(1)の一部として必須です。
寄与の大きさが仮説上のメカニズムと整合しているかも評価してください。
条件数が多い場合、多重比較の補正が必要かどうか、必要なら考慮されているかも確認してください。

出力は次の形式で厳密に返してください:
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容(該当ファイル・箇所・何が期待と違うか)
(指摘がない場合は「指摘: なし」と明記)
```

### 統計だけでなく、出力そのものを見る

寄与率の数字が「妥当そう」に見えても、それだけでは要素の効果が本当に意図通りに現れているかは
分からない。要素を外した際の生成物・予測例・エラー分析の代表例(SKILL.md の「3. 分析」で
集めるもの)を実際に見て、数字の裏付けとして矛盾がないかを確認する。

- 実行主体(自分)は、フル手法と各アブレーション条件について、代表的な出力サンプル(良さそうな
  例・悪そうな例・エラー分析でカテゴリ化した失敗例を含む)を `research/figures/samples/` 等に
  用意する
- 独立したサブエージェントに、**まず寄与率の数値を見せずに**、目的・要素の役割(`selected-method.md`)
  とサンプル出力のパスだけを渡し、「フル手法とその要素を外した条件で、出力に見た目上の違いが
  期待通り現れているか」を判定させる。画像等は Read ツールで直接開けるようパスをそのまま渡す
- そのうえで寄与率の数値・結論も見せ、**見た目の判断と数値上の結論が一致するか**を確認させる。
  一致しない場合(数値は差があるのに見た目はほぼ同じ、その逆)は、寄与率の計算・サンプル選定に
  問題がないかを疑う
- 出力に直接見られる artifact が無いタスク(スカラー指標のみ等)の場合はこの評価は該当なしと
  してよい

**プロンプト例(出力そのものの評価)**:

```
あなたはこの実験の実施には関与していない第三者のレビュアーです。
まだ寄与率の数値は見せません。以下の要素の役割と、フル手法/アブレーション条件それぞれの
出力サンプルを見て、要素を外した効果が見た目上も現れているかを判断してください。

要素の役割: {selected-method.md のパス}
出力サンプル: {フル手法・各アブレーション条件の代表的な出力サンプルのパス一覧}

判断してください:
1. 要素を外した条件は、フル手法と比べて見た目にも違いが分かるか(分かるべき差が本当に見えるか)
2. 見た目の質が疑わしい兆候(意図しない劣化、逆に何も変わっていないように見える等)はないか

出力は次の形式で厳密に返してください:
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容
(指摘がない場合は「指摘: なし」と明記)
```

### 指摘が出た場合

「効いていない」結果の原因分析の順番(実装 → テストデータ → 環境 → 先行研究の認識齟齬 →
手法そのもの)に沿って、判明した原因ごとに対応する:

1. **実装バグ・ライブラリ/APIの誤用 → 前工程(実装)に戻って直す。** 対象コードが
   `research-08-implementation-execution` で扱ったコアな実装(architecture.md のモジュール)
   であれば、そちらの独立レビュー(`research-08-implementation-execution/references/code-review.md`)
   を使って直す。このスキル自身が書いたアブレーション用のコード(要素の切り替え・寄与率計算
   など)であれば、その場で直して**再実行し、この結果の妥当性レビューをやり直す**
2. **テストデータの不整合(座標系・単位・時間軸など) → データの取得・前処理コードを直す**
3. **環境要因(他プロセス・熱スロットリングなど) → 環境を切り分けて再計測する**
4. **先行研究に対する認識齟齬 → 元論文・`selected-method.md` を読み直し、理解を修正した上で
   実装または設計を見直す。** 修正が設計レベルに及ぶ場合は `research-06-algorithm-design` に
   差し戻す
5. **手法そのものの性質(寄与が実際に小さい) → `ablation-study.md` に正直に反映する**
   (寄与が小さい要素を隠さず報告するという `research-10-ablation-study` の品質基準そのもの)。
   ここに至るのは1〜4を全て確認・棄却した後のみ

いずれの場合も、修正後は必ず実データで再実行してから結論を確定する。見た目の判断と数値上の
結論が食い違う場合も、寄与率の計算・サンプル選定を見直し、数値だけで結論を確定しない。

## 記録

レビュー結果は、判定(PASS/FAIL)・指摘の要約・対応を1エントリだけ短く
`research/code-review-log.md` に残す(仕様書ではないので詳細な議事録は不要)。指摘なしで
一発 PASS した場合も「PASS(指摘なし)」の1行は残す。デバッガーが介入した場合は
`ROOT_CAUSE`/`CATEGORY`/`FIX_PLAN` の要約も残す。

他のタスクにも関わる横断的な学び(例:「この比較関数はNaNが混じると常にFalseを返す」)が
見つかった場合は、`code-review-log.md` だけに留めず、`research/tasks.md` の
`## Implementation Notes` にも転記する(`research-08-implementation-execution` 参照)。後続タスクの
実装者はここを読んでから着手するため、同じ落とし穴を踏まずに済む。

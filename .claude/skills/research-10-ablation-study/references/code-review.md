# 実験コードの独立レビュー(3段階)

アブレーション・感度分析のコードは、評価実験のコードを流用・改造して作ることが多く、
「要素を外す/置き換える」の実装ミス(実は外し忘れている、比較対象が評価実験と条件が
ズレている等)が結果を静かに歪める。これは実装した本人(=このスキルを実行している自分)が
読み返しても見落としやすい。そのため、実験に使うコードを書いた・変更した際は、
**過去の会話コンテキストを渡さない独立したサブエージェント**にレビューさせ、
指摘があれば直してから先に進む。

**目次**: [共通ルール](#共通ルール)(変更粒度・実行方法・出力フォーマット・指摘への対応・
デバッガー・完了宣言前のフレッシュな証拠)/ [Stage 1](#stage-1-実装中レビュー仕様適合バグ)
(設計適合・境界チェック)/ [Stage 2](#stage-2-完成時レビューダミーデータでの妥当性検証)
(ダミーデータ・RED-phase証拠)/ [Stage 3](#stage-3-実データ適用後レビュー結果の妥当性)
(結果解釈・出力そのものの目視評価)/ [記録](#記録)

## 共通ルール

### 対象とする変更の粒度

レビュー対象は「結果を左右するコード」に限る: アブレーション条件の実装(要素を外す/
置き換える処理)、感度分析、効率計測、寄与率計算など。ログ文言の変更やコメント修正など
結果に影響しない変更は対象外でよい。全ての瑣末な変更にフルの3段階を毎回回すと、実験の
イテレーション速度を損なうコストに見合わなくなる。目安として、Stage 1 は上記コアロジックを
新規作成・変更するたびに、Stage 2/3 は「実データに投入する前」「結果が出て結論を確定する前」
という節目で1回でよい。

### レビューの実行方法

- レビューは Agent tool でサブエージェント(`general-purpose` 想定)を起動して行う。
- サブエージェントには、それまでの会話・設計判断の経緯を渡さない。渡すのは
  「参照すべき仕様書のパス」「対象コードのパス」「レビューさせたい観点」のみ。
- **モデル**: 実装者(このスキル自身)と同格のモデルでレビューすると、同種の見落としを
  再生産しやすい。Agent tool の `model` パラメータで明示的に指定する。Stage 1・2 は `sonnet`
  以上、判断の難度が最も高い Stage 3(結果解釈の妥当性)は既定で `opus` を使う。
- プロンプトに「これで合っているはず」「一応確認して」のような期待値・誘導を書かない。
  独立した第三者として虚心にジャッジさせる。

### 出力フォーマット(構造化)

サブエージェントには自由記述ではなく、次の形式で厳密に返させる。曖昧な文章判定は
親エージェント側での見落とし・誤読を招くため、機械的に次の一手を判断できる形にする
(下記の各プロンプト例には、この指定をそのまま含めてある):

```
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容(該当ファイル・箇所・何が期待と違うか)
(指摘がない場合は「指摘: なし」と明記)
```

### 指摘への対応

- 重大度「高」(結論を覆しうるもの: 要素の外し方の誤り、条件の不揃い、寄与率計算の誤りなど)は
  必ず解消してから次に進む。「中」「低」は対応するか、対応せず記録だけして先に進むかを
  自分で判断してよい。
- 指摘を解消したら**同じ段階のレビューをやり直す**。指摘を読んだだけで次の段階に進まない。
- **レビュアーの判定は絶対ではない。** 指摘の根拠が薄い、または明確に反証できる場合は、
  盲目的に修正せず、指摘とその反証をユーザーに提示して判断を仰ぐ。
- **同じ指摘で2回連続 reject されたら、闇雲に3回目の修正を試みない。** 代わりに
  「デバッガー」役の独立したサブエージェントを起動する(過去の会話コンテキストを渡さない。
  原因調査に必要なら WebSearch/WebFetch の使用も許可する)。デバッガーには、指摘内容・
  これまで試した修正・関連コードを渡し、次を返させる:
  - `ROOT_CAUSE`: 何が根本原因か
  - `CATEGORY`: 実装バグ/仕様理解のズレ/テスト自体の誤り/**手法自体の論理の誤り(構成要素の
    分解や寄与の前提の問題)**、など
  - `FIX_PLAN`: 次に何をどう直すべきかの具体的な手順
  この fix plan に沿って修正し、レビューをやり直す。**デバッガーのラウンドは最大2回まで。**
  2回のデバッグでも同じ指摘が解消しない場合はレビューを打ち切り、ユーザーに状況(指摘内容・
  試した修正・デバッガーの調査結果)を報告して判断を仰ぐ。無限にレビュー→修正を繰り返さない。
  特に `CATEGORY` が「手法自体の論理の誤り」に近いと判断された場合は、実装を直し続けるのではなく
  `research-04-method-design`・`research-06-algorithm-design` への差し戻しをユーザーに提案する。

### 完了宣言の前に(フレッシュな証拠)

「直った」「うまくいった」「PASS した」とは、記憶や以前の実行結果だけで言わない。必ず
その場で再実行し、新しく得られた証拠(出力・ログ・テスト結果)を確認してから宣言する。
修正を加えた直後に「これで直ったはず」で終わらせず、実際に再実行して確認できるまでは
未検証(NOT_VERIFIED)として扱い、ユーザーにも「未検証」と「検証済み」を混同させない。

## Stage 1: 実装中レビュー(仕様適合・バグ)

**いつ**: アブレーション条件(要素を外す/置き換えるコード)、感度分析、効率計測、
寄与率計算など、結果を左右するコードを書いた直後、実行する前。

**渡す情報**: 対象コードのパス、`research/selected-method.md`(構成要素の分解)、
`research/evaluation-report.md`・`research/implementation-plan.md`。

**観点**:
- 「外す/置き換える」対象の要素が仕様(selected-method.md の構成要素分解)と一致しているか。
  意図した要素だけが変更されており、他の要素やハイパーパラメータが巻き込みで変わっていないか。
- フル手法・各アブレーション条件・ベースラインが評価実験(research-09)と同一プロトコル
  (同じデータ分割・同じシード集合・同じ評価コード)になっているか
- 寄与率・感度分析の計算式が定義通りか(差分の向き、正規化の有無)
- shape 不整合、NaN/Inf、例外の握りつぶし
- **設計との対応関係を項目ごとに確認する。** 「全体として大体合っていそう」という印象だけの
  判断は見落としを生みやすい。設計書(selected-method.md の構成要素一覧)を1項目ずつ拾い、
  対応するコード上の場所(ファイル・関数/クラス名)が実在するかを確認させる。**設計書にある
  のに実装されていない要素**、逆に**設計書にない独自の処理**がないかを洗い出す
- **担当外のモジュールへの越境がないか。** architecture.md のモジュール分割で、このコードが
  変更してよい範囲は決まっているはずである。その範囲を超えて他のモジュール・他のコンポーネントの
  責務にまで手を入れていないか(意図しない副作用、本来疎結合であるべき箇所の密結合化)を確認する

**機械的に確認できることは読解より先にコマンドで確認する。** 印象だけのレビューを補うため:
- `git diff` で実際の変更差分を見る(最終状態のコードだけを読んで「大体合っていそう」と
  判断しない。差分は「何を変えたか」の一次情報であり、意図しないファイルへの変更・消し忘れ・
  境界を越えた変更はここで初めて見えることが多い)
- `TODO` / `FIXME` / `pass` だけのスタブ実装が残っていないか grep する
- 該当するテストを実際に実行し、通ることを確認する(通ったという報告を鵜呑みにしない)

**プロンプト例**(そのまま流用可、パスは実際のものに置換):

```
あなたはこのコードの作成には関与していない第三者のレビュアーです。
以下の設計書とコードを読み、コードが設計書の意図通りに実装されているか、
また典型的なバグ(意図しない要素まで変更してしまっている、評価実験と条件が
ズレている、寄与率の計算ミス、shape不整合等)がないかを客観的にレビューしてください。

設計書: {selected-method.md, evaluation-report.md のパス}
コード: {対象ファイルのパス}

「動くはず」ではなく「仕様と一致しているか」の観点で判断してください。
設計書に書かれている構成要素を1つずつ拾い、コード上の対応箇所が実在するかを確認してください。
設計書にあるのに実装されていない要素、逆に設計書にない独自の処理を追加してしまっている箇所が
あれば指摘してください。

出力は次の形式で厳密に返してください:
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容(該当ファイル・箇所・何が期待と違うか)
(指摘がない場合は「指摘: なし」と明記)
```

## Stage 2: 完成時レビュー(ダミーデータでの妥当性検証)

**いつ**: コードが一通り完成し、実データにかける前。

**やること**(実行主体である自分が行う):

1. 仕様(`selected-method.md`)から、性質の異なる2種類のダミーテストセットを作る。
   - **成功するはずのケース**: ある要素を外すと明確に性能が落ちる/効率が変わることが
     分かっている合成設定(例: その要素がないと解けない小規模な人工タスク)。
   - **絶対に差が出ない・失敗するはずのケース**: 寄与しないはずのダミー要素を外す、
     フル手法と同一条件を2回比較する(差が出てはいけない)など。
2. **RED-phase の証拠を残す。** 寄与率・感度分析の判定ロジックを実装する前(またはスタブ化
   した状態)で両ケースを実行し、「期待通りの判定にならない」ことを一度確認してログに残す。
   これを飛ばして最初からテストがパスしている場合、そのテストは実は何も検証していない
   (何を入れても通ってしまう)可能性がある。これが RED→GREEN の RED にあたる
3. 判定ロジックを実装し、両ケースで実際にコードを実行して、今度は期待通りの判定になる
   (GREEN)ことを確認する。出力・ログを残す
4. 独立したサブエージェントに、コード・両ケースの入力と出力(RED 時と GREEN 時の両方)を
   渡し判定させる。

**観点**:
- 「差が出るはず」のケースで、本当に期待通りの寄与(性能低下・効率変化)が検出できているか。
- 「差が出ないはず」のケースで、コードが正しく「差がない」と判定できているか。
  逆に、バグにより見かけ上「寄与あり」と誤って検出していないか(これが最も見落とされやすい)。
- 寄与率・感度分析の判定基準(どこからを「寄与あり」とするか)自体が仕様と整合しているか。
- **テストケース自体の設計が妥当か。** 閾値が緩すぎてバグがあっても通ってしまわないか、
  差が出るはずのケースの効果が弱すぎて/強すぎて検証になっていないかも合わせて判定させる
  (テストケースを作った本人にはこの甘さが見えにくい)。
- **RED-phase の証拠があるか。** ロジック実装前(またはスタブ状態)で実行して「期待通り
  にならない」ことを確認したログが無い場合、そのテストが本当にバグを検出できる設計かどうか
  誰も確認していないことになる。RED の証拠が無ければ、その旨を指摘する。

**プロンプト例**:

```
あなたはこのコードの作成には関与していない第三者のレビュアーです。
以下は実験コードに対して作成した2種類のテストケースと、その実行結果です。

コード: {対象ファイルのパス}
ケースA(明確な寄与があるはずの合成設定とその結果): {パス・出力}
ケースB(寄与がない/差が出ないはずの設定とその結果): {パス・出力}
RED-phase の証拠(ロジック実装前・スタブ状態での実行結果): {あれば、そのログ}
判定基準の仕様: {selected-method.md 等のパス}

以下を判定してください:
1. ケースAは本当に「寄与あり」と判定されるべき結果になっているか
2. ケースBは正しく「寄与なし/差がない」と判定されているか。バグにより誤って
   「寄与あり」と判定されていないか
3. 寄与率・感度分析の判定ロジック自体が仕様と整合しているか
4. ケースA・ケースBのテストケース自体の設計は妥当か(閾値が緩すぎてバグを見逃さないか、
   効果の強さが検証にふさわしいか)
5. RED-phase の証拠が示されているか。無い場合、テストが本当にバグを検出できる設計か
   どうか未確認であることを指摘する

出力は次の形式で厳密に返してください:
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容(該当ファイル・箇所・何が期待と違うか)
(指摘がない場合は「指摘: なし」と明記)
```

指摘があれば、コードを修正して **Stage 1 からやり直す**(判定ロジックの修正はコード変更を伴うため)。

## Stage 3: 実データ適用後レビュー(結果の妥当性)

**いつ**: 実データでアブレーション・感度分析を実行し、結果が出た後、`ablation-study.md`・
`sensitivity-analysis.md` の結論を確定する前。

**渡す情報**: `research/selected-method.md`、条件表(事前登録した比較条件)、
`research/experiment-results.md`(生データ)、`ablation-study.md`/`sensitivity-analysis.md` のドラフト。

**観点**:
- 「要素 X が効いている」という結論の場合: それが本物の寄与によるものか、それとも
  条件の揃え忘れ・シード数不足・比較コードのバグで説明できてしまわないか。
- 「要素 X は効いていない」という結論の場合: **まず実装バグを疑う。** 「寄与が本当に小さい」
  よりも「外したつもりで外れていない」等のバグの方が多い。寄与が小さいと結論する前に、
  **最も単純なケース**で改めて実行し、そこで期待通りの差が出るかを再評価する。ここでいう
  「単純」とは、差が最も大きく出るように設計したケース(Stage 2 の「差が出るはずのケース」は
  効果検出を狙って作られており、必ずしも単純とは限らない)ではなく、**手計算・目視で正解が
  分かるくらい小さく作った入力**を指す。最も単純なケースですら差が出ない場合はほぼ確実に
  実装側の問題
- 寄与の大きさが、下がり方のメカニズム(仮説)と整合しているか、それとも数字だけ見て
  こじつけていないか。
- **多重比較の補正。** アブレーションは条件数が多くなりがちで、条件ごとに有意差検定を
  行うと第一種の過誤(偶然の差を実際の寄与と誤認するリスク)が積み上がる。Bonferroni 補正等が
  必要か、必要なら考慮されているかを確認する。
- 好都合な条件の結果だけを見て、都合の悪い結果(相互作用が疑われる組み合わせなど)を
  無視していないか。

**プロンプト例**:

```
あなたはこの実験の実施には関与していない第三者のレビュアーです。
以下の構成要素の分解・比較条件・生の実験結果・導かれた結論を読み、結論の妥当性を検証してください。

構成要素と比較条件: {selected-method.md, 条件表}
生データ: {experiment-results.md}
導かれた結論: {ablation-study.md / sensitivity-analysis.md の該当部分}

「要素Xが効いている/効いていない」という結論が、比較条件の不揃い・シード数不足・
実装のバグなど別の要因で説明できてしまわないか検討してください。「効いていない」結論であれば、
まず実装バグの可能性を疑い、最も単純なケース(差が最も大きく出るケースではなく、手計算・
目視で正解が分かるくらい小さい入力)で改めて妥当な差が出るか確認したかを問い、確認していなければ
その点を指摘してください。
寄与の大きさが仮説上のメカニズムと整合しているかも評価してください。
条件数が多い場合、多重比較の補正が必要かどうか、必要なら考慮されているかも確認してください。

出力は次の形式で厳密に返してください:
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容(該当ファイル・箇所・何が期待と違うか)
(指摘がない場合は「指摘: なし」と明記)
```

### 統計だけでなく、出力そのものを見る

寄与率の数字が「妥当そう」に見えても、それだけでは要素の効果が本当に意図通りに現れているかは
分からない。要素を外した際の生成物・予測例・エラー分析の代表例(SKILL.md の「3. 分析」で
集めるもの)を実際に見て、数字の裏付けとして矛盾がないかを確認する。

- 実行主体(自分)は、フル手法と各アブレーション条件について、代表的な出力サンプル(良さそうな
  例・悪そうな例・エラー分析でカテゴリ化した失敗例を含む)を `research/figures/samples/` 等に
  用意する
- 独立したサブエージェントに、**まず寄与率の数値を見せずに**、目的・要素の役割(`selected-method.md`)
  とサンプル出力のパスだけを渡し、「フル手法とその要素を外した条件で、出力に見た目上の違いが
  期待通り現れているか」を判定させる。画像等は Read ツールで直接開けるようパスをそのまま渡す
- そのうえで寄与率の数値・結論も見せ、**見た目の判断と数値上の結論が一致するか**を確認させる。
  一致しない場合(数値は差があるのに見た目はほぼ同じ、その逆)は、寄与率の計算・サンプル選定に
  問題がないかを疑う
- 出力に直接見られる artifact が無いタスク(スカラー指標のみ等)の場合はこの評価は該当なしと
  してよい

**プロンプト例(出力そのものの評価)**:

```
あなたはこの実験の実施には関与していない第三者のレビュアーです。
まだ寄与率の数値は見せません。以下の要素の役割と、フル手法/アブレーション条件それぞれの
出力サンプルを見て、要素を外した効果が見た目上も現れているかを判断してください。

要素の役割: {selected-method.md のパス}
出力サンプル: {フル手法・各アブレーション条件の代表的な出力サンプルのパス一覧}

判断してください:
1. 要素を外した条件は、フル手法と比べて見た目にも違いが分かるか(分かるべき差が本当に見えるか)
2. 見た目の質が疑わしい兆候(意図しない劣化、逆に何も変わっていないように見える等)はないか

出力は次の形式で厳密に返してください:
判定: PASS または FAIL
指摘:
- [重大度: 高/中/低] 指摘内容
(指摘がない場合は「指摘: なし」と明記)
```

指摘が出た場合:
- コードの問題が疑われる → 該当箇所を修正し **Stage 1 からやり直す**(実験の再実行が必要になる)。
- 見た目の判断と数値上の結論が食い違う → 寄与率の計算・サンプル選定を見直し、数値だけで
  結論を確定しない
- 「寄与が小さい/効いていない」結論で、最も単純なケースでの再評価をまだ行っていない →
  先に手計算・目視で正解が分かる規模の最も単純なケース(Stage 2 の「差が出るはずのケース」を
  そのまま使うのではなく、必要ならさらに単純化する)で妥当な差が出るか確認してから、
  寄与が実際に小さいのかを判断する
- 単純なケースでも妥当な差が出ないことを確認した上で、なお手法自体の性質(寄与が実際に小さい等)
  と判断される → `ablation-study.md` に正直に反映する(寄与が小さい要素を隠さず報告するという
  `research-10-ablation-study` の品質基準そのもの)。

## 記録

各段階のレビュー結果は、判定(PASS/FAIL)・指摘の要約・対応を1エントリだけ短く
`research/code-review-log.md` に残す(仕様書ではないので詳細な議事録は不要)。
指摘なしで一発 PASS した場合も「Stage n: PASS(指摘なし)」の1行は残す。デバッガーが
介入した場合は `ROOT_CAUSE`/`CATEGORY`/`FIX_PLAN` の要約も残す。
再現性フェーズ(`research-11-reproducibility`)で「独立レビューを実施済みか」を
確認する際の材料になる。

他のタスクにも関わる横断的な学び(例:「この比較関数はNaNが混じると常にFalseを返す」)が
見つかった場合は、`code-review-log.md` だけに留めず、`research/tasks.md` の
`## Implementation Notes` にも転記する(`research-08-implementation-execution` 参照)。後続タスクの
実装者はここを読んでから着手するため、同じ落とし穴を踏まずに済む。

# アブレーション研究(Ablation Study)

提案手法の各構成要素の寄与を分離して評価し、「なぜ効くのか」に証拠を与える。

## 前提

開始時に `research/STATUS.md`・`research/selected-method.md`(構成要素の分解)・`research/evaluation-report.md`・`research/experiment-results.md`・`research/tasks.md`(あれば)を読む。
`tasks.md` があれば `## Implementation Notes` に既知のハマりどころが無いか確認する。
評価実験と同じデータ分割・シード・評価コードを使う(条件を変えると比較にならない)。

## 進め方

### 1. アブレーション設計

- selected-method.md の構成要素ごとに「外す/置き換える」実験を列挙する
- 各実験に問いを 1 つ対応させる: 「要素 X を外すと指標がどれだけ落ちるか → X の寄与」
- 実行コストを見積もり、優先順位を付けてユーザーに確認する(全部やると高くつく場合は主要要素に絞る)

### 2. 実施

自分で書いたコードは、動けば正しいと思い込みやすい。「要素を外したつもりが外れていない」
「比較条件が評価実験とズレている」といった誤りは実行時エラーにならず、そのまま寄与率の
数字として結果に紛れ込む。実装者本人の見直しだけでは防ぎにくいため、過去の会話コンテキストを
渡さない独立したサブエージェントに客観的にレビューさせる。手順・プロンプト例は
`references/code-review.md` を参照。

**「できる限り単純な疑似データ → 実データ」の順に評価し、うまくいかなければ実装に戻って
疑似データ評価からやり直す、というループで手法の論理的な品質と実装品質の両方を上げていく**
(research-09-evaluation と同じ考え方)。

- 評価実験と同じ事前登録スタイル: 実行前に条件表を確定する
- アブレーション条件のコード(要素を外す/置き換える処理、寄与率計算)を書いた/変更した直後、
  実行する前に **`references/code-review.md` の Stage 1(設計適合・バグの独立レビュー)** を通す。
  「大体合っていそう」ではなく、設計書の構成要素とコードの対応を1つずつ確認させる
- コードが完成し実データに投入する前に、**`references/code-review.md` の Stage 2**
  (明確に寄与があるはずのダミー設定/差が出ないはずのダミー設定での判定ロジックの妥当性レビュー)
  を通す。指摘があればコードを直し、Stage 1 からやり直す。**このループを、疑似データで
  PASS するまで実データには進まない**
- **同じ疑似データケースが、実装を直しても(デバッガーを挟んでも)繰り返し失敗する場合、
  実装のバグではなく手法自体の論理(構成要素の分解や寄与の前提)に問題がある可能性を疑う。**
  その場合は実装を直し続けるのではなく、`research-04-method-design`・`research-06-algorithm-design`
  に戻って手法自体を見直す選択肢をユーザーに提示する
- フル手法・各アブレーション条件・ベースラインを同一プロトコル(同シード集合)で比較
- 複数条件を並行実行する場合は、各条件の出力先を個別ファイルにする(research-09-evaluation の
  実施要領と同じ)。`research/experiment-results.md` への追記は、各実行の完了を確認したうえで
  本体(自分自身)が単独で行い、複数プロセスが同時に同じファイルへ書き込むことは避ける

### 3. 分析

- **寄与率分析**: 要素ごとの性能差分の表。相互作用が疑われる組み合わせは 2 要素同時除去も試す
- **パラメータ感度分析**: 主要ハイパーパラメータを振って性能の頑健性を確認(結果が 1 点のチューニングに依存していないか)
- **効率分析**: 学習曲線(収束速度)、推論時間、メモリ使用量をベースラインと比較
- **エラー分析**: 失敗事例を目視でカテゴリ分けし、各カテゴリの件数と代表例を記録。「どこで失敗するか」は限界の記述と将来研究の根拠になる
- **ボトルネック分析**: 性能・速度の律速要因を特定
- **結論を確定する前に `references/code-review.md` の Stage 3(実データ適用後・結果の妥当性の
  独立レビュー)を通す。** 「要素Xが効いている/効いていない」という結論が、比較条件の不揃い・
  実装のバグで説明できてしまわないかを独立したサブエージェントに検証させる。コードの問題が
  疑われれば直して Stage 1 からやり直し、**疑似データでの評価(Stage 2)を再度 PASS させてから
  実データに戻る**(実装を直した勢いで実データにいきなり再挑戦しない)。寄与が実際に小さい
  だけであれば正直に報告する
- **「効いていない」場合はまず実装バグを疑う。** 寄与が小さいと結論する前に、最も単純な
  ケース(差が最も大きく出るケースではなく、手計算・目視で正解が分かるくらい小さい入力)で
  改めて妥当な差が出るかを再評価する
- **寄与率の数字だけで判断しない。** フル手法とアブレーション条件の代表的な出力サンプルを
  `research/figures/samples/` に保存し、Stage 3 の独立レビューで数値を見せる前にまず出力の
  見た目の違いが期待通り現れているかを判定させる(詳細は `references/code-review.md` を参照)

### 4. 出力

```
research/
    ablation-study.md        # アブレーション条件表、結果、寄与率分析、エラー分析、結論
    sensitivity-analysis.md  # ハイパーパラメータ感度、効率分析(時間・メモリ)、頑健性の総括
    figures/samples/         # 代表的な出力サンプル(独立レビューでの目視評価用)
```

### 5. 完了処理

- `research/STATUS.md` を更新(主要な寄与要素を1行で記録)
- 結果をユーザーに確認してもらう際は、グラフ・出力サンプルを埋め込んだ HTML でまとめて提示する
  (Artifact tool が使える環境ならそれで公開し、無ければファイルに保存してパスを案内する)。
  ファイルパスや数値の羅列だけでは、図の内容が伝わらず確認の質が落ちる
- 次のステップ `/research-11-reproducibility` を案内

## 品質基準

- 実験コードは自己レビューだけで進めない(`references/code-review.md` の3段階レビューを飛ばさない)
- 「効いていない」結果を見たら、まず実装バグを疑う。最も単純なケースで妥当な差が出るかを
  確認してから、寄与が実際に小さいのかを判断する
- 疑似データ → 実データの順で評価し、うまくいかなければ実装を直して疑似データ評価からやり直す
  (実データにいきなり戻らない)。同じ疑似データケースで繰り返し行き詰まる場合は、実装では
  なく手法の論理自体を疑い、method-design/algorithm-design への差し戻しを検討する
- 寄与率の数字だけで判断しない。代表的な出力サンプルを実際に見て(独立レビューさせて)
  効果が見た目上も期待通り現れているかを確認する
- 「要素を外したら下がった」だけでなく、下がり方が仮説のメカニズムと整合しているかを議論する
- 寄与が小さい要素は正直に報告する(手法の簡素化提案につながる — それも貢献)
- アブレーションの表は論文の Ablation セクションにそのまま使える形式で作る

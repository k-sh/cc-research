# 研究計画の策定

研究テーマを明確化し、以降の全工程(調査→設計→実装→評価→論文化→公開)の土台となる研究計画を作成する。

## 進め方

### 1. ヒアリング

ユーザーの最初の説明だけで計画を書き始めない。曖昧さが残る項目は AskUserQuestion でまとめて確認する。確認すべき項目:

- **研究目的**: 何を明らかにしたい/実現したいのか。「なぜそれをやりたいのか」を一段抽象化して聞き、本当の目的(実務課題の解決か、学術的貢献か、両方か)を特定する
- **研究分野**: 分野・サブ分野。学会/ジャーナル投稿を想定するならターゲット
- **制約**: 期間、計算資源(GPU 有無・種類)、データ(既存/要収集、ライセンス)、予算、人員、秘密保持
- **実験環境**: 使用機器、OS、既存コードベースの有無
- **評価指標**: 成功をどう測るか。定量指標(精度、速度、メモリ等)と目標値
- **想定成果物**: 論文、OSS、プロトタイプ、特許、社内レポートなど
- **スコープ**: やること/やらないことの境界

すでに会話やリポジトリに情報がある項目は聞き直さない。

### 2. 文書化

以下を作成する。既存ファイルがあれば上書きせず差分を確認してから更新する。

```
research/
    objective.md           # 研究目的(背景・動機・目的の抽象化・具体化・成功基準)
    constraints.md         # 制約事項(資源・期間・データ・法務・その他前提)
    evaluation-metrics.md  # 評価指標(指標の定義・測定方法・目標値・ベースラインとの比較方法)
    research-plan.md       # 研究計画(スコープ・調査キーワード・フェーズ計画・リスク)
```

`research-plan.md` には必ず含める:

- **調査キーワード**: 日本語と英語の両方。同義語・関連語も列挙(次の literature-review が使う)
- **フェーズ計画**: 各フェーズ(調査/設計/実装/評価/執筆)の目安期間と完了条件
- **リスクと代替案**: 主要な不確実性と、うまくいかなかった場合のフォールバック
- **仮のリサーチクエスチョン**: 現時点での問い(gap-analysis 後に更新される前提で書く)

### 3. STATUS.md の初期化

`research/STATUS.md` を作成する:

```markdown
# 研究進行状況

## テーマ
(1行)

## 完了フェーズ
- [x] 01 planning (YYYY-MM-DD) — 主要決定: ...
- [ ] 02 literature-review
- [ ] 03 gap-analysis
- [ ] 04 method-design
- [ ] 05 patent-search
- [ ] 06 algorithm-design
- [ ] 07 implementation
- [ ] 08 evaluation
- [ ] 09 ablation-study
- [ ] 10 reproducibility
- [ ] 11 paper-writing
- [ ] 12 slide-generation
- [ ] 13 review-response
- [ ] 14 release-preparation

## 未解決の論点
- ...
```

### 4. 完了報告

作成した文書の要点(目的・主要制約・評価指標・調査キーワード)を要約して提示し、
次のステップとして `/research-02-literature-review` を案内する。

## 品質基準

- 成功基準は測定可能な形で書く(「精度を上げる」ではなく「データセット X で指標 Y をベースライン比 Z% 改善」)
- 自分が置いた仮定は「仮定」と明示し、ユーザー発言と区別する
- 制約は後工程の判断材料になるため、曖昧なものも「未確定」として記録する
